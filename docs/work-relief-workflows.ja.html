AI Automation Starter Kit

作業負担軽減5パック: メール送信なしのローカル確認フロー

この手順書は、業務引き継ぎと期限確認、入社・異動・退職手続き確認、契約書・申込書の完備確認、見積比較と購入前確認、監査・許認可の期限確認の5パックを対象にしています。すべてローカルのフォルダ、ローカルUI、PIN承認で完結します。メール自動化は対象外です。

ローカル完結

資料確認、下書き、承認、監査記録は作業端末内で完結します。

最後は人が決める

Codex は質問と下書きを担当し、承認者が内容確認と PIN 承認を担当します。

メール送信は対象外

Gmail とメール自動化はこの5パックの対象外で、別プロジェクトとして扱います。

  • 最初に読む
  • 導入とログイン
  • パック選択
  • フォルダ入力
  • 画面操作
  • 翌日の再利用
  • 困ったとき
  • 安全境界
  • 副業としての納品
  • 英語版

1. このルートが向いている仕事

最初に境界を確認してください

この5パックは、繰り返し発生する確認作業を軽くするためのものです。メール送信、Gmail 更新、顧客返信、法的提出、人事記録の変更、購買確定、外部システム更新は行いません。Gmail とメール自動化は別プロジェクトです。

向いている例

毎日または頻繁に同じ種類の資料が届き、下書きや確認票の形が決まっていて、責任者が最後に確認できる仕事です。

向いていない例

受信トレイを常時監視する仕事、メール送信が本体の仕事、顧客への約束を含む仕事、法務提出や給与反映のような不可逆処理です。

得られるもの

ローカル作業場所、パック別の確認フロー、AI からの確認質問、下書き、PIN 承認記録、翌日の準備済みフォルダです。

python3 -m venv .venv

道具の準備を確認する

codex login status
対応環境の事実

このローカル作業場所の安全な作成と承認更新は、macOS または Linux を前提にしています。

ログインの事実

codex login status が未完了なら、そこで止めて先にログインを完了してください。この手順で API キー方式へ切り替える必要はありません。

Codex への頼み方

月報用の案内は Codexではじめる にあります。この5パックは、そのローカル確認ループを踏まえて使います。

対象外の資料を混ぜないでください

この5パックはファイルベースで、メール送信なしの確認フローです。受信トレイの生データを置いて、そのまま返信や送信までやってくれる前提で使わないでください。Gmail とメール自動化は別プロジェクトです。

5. ローカル UI で進める順番

ローカル UI を起動する 
ai-automation-kit office-workspace serve \
  --root "$HOME/AI-Office-Workspaces" \
  --language ja
1. 作業場所を作る

パックを選び、最初の日付を YYYY-MM-DD で入力し、作業場所名、承認者名、ローカル承認 PIN を設定します。

2. フォルダを開く

フォルダ一覧から 01_APPROVED_PAST_OUTPUTS、02_PAST_SUPPORTING_FILES、03_CURRENT_INPUTS/<日付> を開き、置き場を見えるようにします。

3. 資料を確認する

資料を確認 を押し、受け付けた資料、却下された資料、対象外になった資料を確認してから次へ進みます。

4. AI の質問に答える

資料だけでは決められないルールや例外条件だけを質問されます。曖昧なまま進めないでください。

5. 下書きを作る

下書きを作成 を押すと、05_DRAFTS/<日付> に確認用の下書きが出ます。

6. 内容を確認する

名前、日付、数量、担当、期限、不足証跡、例外扱いを確認し、外部に影響する記述は特に慎重に見ます。

7. PIN で承認する

承認者が名前と PIN を入れると、承認済み完成版が 06_APPROVED_OUTPUTS/<日付> に保存され、監査記録がローカルで更新されます。

パック AI がよく聞く質問 人が丁寧に答えるべき点
引き継ぎ・期限確認 どの遅延を高リスクとみなすか。未解決事項の当日担当は誰か。 顧客に見える期限変更、例外承認、正式エスカレーション
従業員ライフサイクル確認 当日中に終えるべき項目は何か。どの項目で管理者確認が必要か。 人事例外、給与影響、アクセス付与や停止の判断
契約書類・申請書完備確認 必須項目は何か。どこまで揃えば人の審査へ回せるか。 法務解釈、受理、提出、署名の判断
見積比較・購入前確認 承認基準はいくらか。どの欠落が購入確認を止めるか。 仕入先選定、購入判断、価格例外、確約
コンプライアンス・ライセンス期限確認 何日前から緊急扱いにするか。同日エスカレーション対象は何か。 更新承認、外部提出、コンプライアンス承認、法務連携

6. 翌日にそのまま再利用する

次の日付を準備する

承認後に UI から次の日付を準備します。作成前に候補日付を確認してください。

承認済み完成版を見本にする

直前の承認済み完成版は、次回の構成や書きぶりの見本として再利用できます。

当日資料を混ぜない

新しい 03_CURRENT_INPUTS/<日付> には新しい日の資料だけを入れてください。前日分を混ぜないことが大切です。

信頼できる過去完成版の扱い

再利用できるのは同じ作業場所で承認済みになった完成版だけです。次回実行前に記録済みハッシュを再確認し、改変されたファイルは見本として使わずに止まります。

再利用の意味を広げすぎないでください

これは永続的な学習ではありません。次回の隔離された入力に、承認済み見本を参照として渡すだけです。新しい日の正しさは新しい資料で決まります。

7. 困ったときの確認順

Codex の準備ができていない

codex login status を実行し、先にローカルログインを完了してください。この流れで API キーを貼り付ける必要はありません。

UI は開くがパックが使えない

作業場所一覧を更新し、最初の日付が YYYY-MM-DD 形式か、起動した保存先ルート配下に作業場所があるかを確認してください。

資料が却下または対象外になった

拡張子、破損、実際の配置場所を確認してください。当日資料は 03_CURRENT_INPUTS/<日付> に直接置く必要があります。

下書きが薄い、または根拠が足りない

不足資料や固定資料を追加し、再度資料確認を行い、追加質問に答えてから下書きを作り直してください。推測を承認で通さないでください。

承認者が承認できない

承認者名と PIN が作業場所作成時の設定と一致しているか確認してください。承認はこの作業場所とこの端末のローカル記録です。

翌日の見本が正しく出ない

当日分が承認済みか、承認後にファイルが変更されていないかを確認してください。改変済みファイルは次回実行で拒否されます。

途中で実行が止まった

まず状態更新を行ってください。手動復旧が必要なら、00_START_HERE/workspace_status.json、.system/workspace.json、現在周期の状態ファイル、未完了 run、07_AUDIT/audit.jsonl を順に確認します。

8. 安全境界

Codex がしてよいこと

対応形式のローカル資料を読み、事実を整理し、不足ルールを質問し、ローカル下書きを作り、承認済み過去完成版を見本として参照することです。

Codex がしてはいけないこと

メール送信、Gmail 更新、仕入先連絡、期限の本番更新、人事記録変更、外部提出、支払、自己承認です。

人が必ず行うこと

正しい資料選定、業務ルールへの回答、下書き確認、PIN 承認、ローカル外でどう扱うかの判断です。

対象外を明記します

Gmail とメール自動化はこの手順書の対象外で、別プロジェクトとして扱います。この5パックを受信トレイ自動化として説明しないでください。

9. 副業として納品するときの範囲

1. まず1パックだけ売る

入力が明確で、承認者が1人に定まり、確認時間を測りやすいパックから始めます。

2. サンプルはマスク済みで受ける

最初は匿名化済みや承認済みの資料だけでデモを作り、ローカル環境で見せます。

3. AI の止まる場所を見せる

AI が止まるのは下書き、確認票、比較表、期限キューまでだと明言し、その先は顧客側の責任者が持つ形にします。

4. 地味な効果を測る

削減できた確認時間、不足項目率、修正回数、承認までの時間を測ってから次の提案へ進みます。

5. Gmail は別件で切り出す

受信トレイ自動化や提案メール送信まで求められたら、別プロジェクトとして要件、権限、監査、失敗時対応を分けて見積もります。

約束しないこと

完全自動のバックオフィス運用、自動法令順守、自動人事完了、自動購買、自動メール対応は約束しないでください。約束するのは、ローカルで見える確認フローと承認境界です。
