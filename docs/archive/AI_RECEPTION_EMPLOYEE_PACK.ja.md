# AI受付社員パック

AI受付社員パックは、このプロジェクトで最初に提案しやすい業務自動化サービスです。

対象は、AIエージェントにまだ慣れていない人です。いきなり大きなAIプラットフォームを作るのではなく、中小企業の問い合わせ、予約前確認、見積もり前ヒアリング、要確認案件の整理、日次レポートなど、目に見える受付業務から始めます。

このパックの目的は、次の流れを簡単にすることです。

1. AI受付フローを選ぶ。
2. 必要なAPIキー、受付フォルダ、入力データ、出力先を確認する。
3. ローカルでdry-runする。
4. AIが作った下書きと承認キューを確認する。
5. `operator_ui/index.html` で顧客に見せる。
6. 有料dry-run PoCとして提案する。

## 自動化できる業務

AI受付社員パックは、次のような業務に向いています。

- LINE、Webフォーム、メールの問い合わせ受付。
- FAQをもとにした一次返信の下書き。
- 予約前の確認事項の聞き取り。
- 見積もり前に必要な情報の整理。
- Google Sheetsのような受付台帳への記録。
- クレーム、契約、返金、医療・法律・金融の判断など、人間確認が必要な案件のエスカレーション。
- 1日の問い合わせ件数、未対応、要確認をまとめる日次レポート。

ポイントは、AIに勝手に送信・判断させることではありません。人間が確認できる形で、受付業務を見える化し、下書きと整理を自動化することです。

## 用意するもの

実際の顧客に導入する前に、最低限これを用意します。

- 将来使うAPIキーまたはOAuth情報。
- LINE、メール、フォーム、CSV、スプレッドシートなどの受付フォルダまたは入力元。
- 最初のdry-run用の安全なサンプルデータ。
- 作業キュー、返信下書き、承認記録、レポート、ローカル送信箱を書き出す出力フォルダ。
- FAQ、サービスメニュー、返信ルール、問い合わせ項目。
- 人間の承認担当者。
- AIが止まって人間に回す条件。
- 本番運用前の停止条件と戻し方。

この準備が難しい場合は、Claude Code、Codex、CursorなどのAIエージェントに、このGitHubプロジェクトを読ませてください。

使える指示例:

```text
このGitHubプロジェクトを確認して、AI受付フローの導入手順を教えて。
LINE問い合わせが多い小規模店舗向けに、最適なフローを選んで。
docs/AI_RECEPTION_EMPLOYEE_PACK.ja.md をもとに、顧客に依頼する準備物リストを作って。
dry-run前に必要なAPIキー、受付フォルダ、サンプルデータを整理して。
ai-reception-line-inquiry を美容室 / スクール / 士業 / リフォーム会社向けに修正して。
```

## 推奨コマンド

```bash
ai-automation-kit flows install ai-reception-line-inquiry --output .tmp/ai-reception-line-inquiry
```

生成後に確認するファイル:

- `setup_requirements.md`
- `client_setup_request.md`
- `connector_status.md`
- `operator_ui/index.html`
- `monetization_plan.md`
- `workflow_map.mmd`
- `human_approval_points.md`

安全なローカルdry-run:

```bash
cd .tmp/ai-reception-line-inquiry
python3 scripts/run_automation.py
python3 scripts/approve_all.py --approver owner@example.com
python3 -m pytest tests/test_flow_contract.py -q
```

初期状態では、実際のメッセージ送信、顧客システムの更新、本番データへの書き込みは行いません。

## operator_ui/index.html の役割

`operator_ui/index.html` は、顧客に見せるためのローカル画面です。

表示する内容:

- 何を自動化するか。
- 業務フローの各ステップ。
- 人間承認が必要な箇所。
- 成果指標。
- 有料PoC前に確認するファイル。

プログラムに詳しくない人でも、「この仕組みで何ができるのか」を見て理解できることが重要です。

## 有料dry-run PoCとして提案する

最初から完全自動化を売るのではなく、有料dry-run PoCとして提案します。

現実的な範囲:

- 1つの業務フロー。
- 1つの受付フォルダまたは入力元。
- 1つの出力先。
- 1人の承認担当者。
- サンプルデータのみ。
- 3日から5営業日。
- 本番送信なし。
- 取り消せない更新なし。
- 医療、法律、金融、採用、返金、決済、契約判断をAIに任せない。

価格目安:

- 初回dry-run PoC: 5万円から15万円。
- 初期設定と初月支援: 10万円から30万円。
- 承認後の月額保守: 3万円から10万円。

これは収益保証ではありません。収入を約束しないでください。削減効果を断定しないでください。dry-runの証拠を見せ、顧客に継続・修正・停止を判断してもらいます。

## なぜ受付から始めるのか

受付・一次対応は最初の商品に向いています。

- 課題が分かりやすい。
- 問い合わせ漏れの損失が伝わりやすい。
- サンプルデータで見せやすい。
- 返信前に人間が確認できる。
- LINE、フォーム、メール、スプレッドシートが中小企業にも馴染みやすい。
- 危険な判断を人間に回しやすい。

営業アウトバウンド、法務判断、医療助言、金融判断、返金、決済、採用合否などは、最初の自動化商品には向きません。

## 本番運用前の境界

このプロジェクトは、まずセットアップとdry-runのために使います。本番運用前には、次を確認してください。

- 顧客がデータの利用権限を持っている。
- APIキーやOAuthの権限範囲を確認した。
- 人間承認ルールを決めた。
- 禁止アクションを決めた。
- 失敗時の戻し方を決めた。
- サンプルデータでテストした。
- `connector_status.md` と `share-check` を確認した。
- 本番開始について顧客の承認を得た。

このパックの価値は、初心者が顧客に分かりやすく説明し、安全に試し、現実的な有料PoCとして提案できる状態を作ることです。
