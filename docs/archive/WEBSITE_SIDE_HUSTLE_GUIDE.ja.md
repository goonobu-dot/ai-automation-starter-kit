# ホームページ副業ガイド

このガイドは、このリポジトリを「小規模事業者向けホームページ制作の実務スターターキット」として公開・活用したい人向けです。

Codex、Claude Code、Cursor などで使う場合は、あわせて [ホームページ制作プロジェクト AIエージェント利用ガイド](WEBSITE_PROJECT_AGENT_GUIDE.ja.md) も読んでください。

## 目指す形

単に「AIで綺麗なサイトが作れる」では弱いです。

必要なのは次です。

- 顧客ヒアリングの型
- 商用利用を意識した公開情報の整理
- ライセンスとオリジナリティのルール
- AIで設計と実装を進める手順
- デザイナー品質に近づくためのAIエージェント批評ループ
- 予約や問い合わせを集約する裏側の運用
- ブラウザーでのQAチェック
- 提案書、価格表、保守範囲

ここまで揃って、初めて副業スターターキットとして価値が出ます。

## 盛り込む価値が高い公開情報

公開情報は「そのままコピーするため」ではなく、「安全に学んで流用するため」に使います。

- Anthropic Claude Skills / Claude Code Skills: AIエージェントに手順や専門知識を渡す考え方
- OpenAI Codex: コード編集、テスト、レビューを進めるエージェント型開発
- Figma Dev Mode MCP: Figma上のデザイン情報を実装に接続する仕組み
- Vercel v0: UI案を高速に探索するためのAIツール
- `shadcn/ui`: React向けの実用的なUI部品
- `Astro`: 軽量で速い静的サイト向けフレームワーク
- `Headless UI`: アクセシブルなメニューやダイアログ
- `Open Props`: CSSトークン集
- `lucide`: アイコン
- WCAGクイックリファレンス: アクセシビリティ基準
- `web.dev/accessibility`: 実装寄りの解説
- Google Search SEO starter guide: 基本SEO
- `schema.org/Hotel` など: 構造化データの参考

## 売り方

最初から「どんなホームページでも作れます」は広すぎます。

先に絞る方が強いです。

- 観光ホテル・ヴィラ
- クリニック・整体
- 美容室・サロン
- 飲食店
- 士業
- 地域サービス業

業種ごとに、必要なCTA、信頼材料、FAQ、文章の温度が違います。

## 品質を上げるポイント

この副業パックが強くなるのは、初心者が次の失敗を避けられるときです。

1. 競合サイトをそのまま真似する
2. 見た目だけ整っていて、導線や信頼情報が弱い
3. スマホ確認とアクセシビリティ確認を飛ばす
4. コードは作れたが、提案、保守、問い合わせ対応の仕組みがない
5. AIが出した見た目を、批評・ブラウザー確認・人間承認なしで完成扱いする

## AIエージェントでデザイナー品質に近づく考え方

最近の論点は「プログラマーがデザインをやるのか」「デザイナーがプログラムをやるのか」になりがちです。ですが、このプロジェクトでは次の問いに置き換えます。

> AIエージェントによって、デザインの判断手順を初心者でも再現できる形にできるか。

一発のプロンプトで完璧なページを作るのではなく、次のループを回します。

1. ヒアリング: 誰向けか、何を売るか、信頼材料は何か、何を押してほしいか
2. 方向性: 文字、色、余白、写真、セクション順、競合と違う点
3. 実装: 実データに近い文章、フォーム、スマホ対応、CTA配置
4. 批評: 階層、わかりやすさ、信頼、オリジナリティ、アクセシビリティ、スマホ表示
5. 修正: 一番大きいリスクから直し、ブラウザーで再確認

これは高難度のブランディング案件でデザイナーを不要にするものではありません。ただ、プログラマーや初心者がAIを使って「判断理由を説明し、見た目を検証し、改善する」流れを持てるようになります。

## 実務フロー

1. 業種を一つ選ぶ
2. 顧客ヒアリングを行う
3. 公開情報とライセンスを確認する
4. オリジナルの方向性を作る
5. AIエージェントでデザイン批評ループを回す
6. 実装する
7. 予約・問い合わせの台帳、ステータス、返信テンプレートを作る
8. PC、スマホ、フォーム、裏側の管理画面を確認する
9. 提案書と保守案を添えて納品する

## AIに任せる部分と人が残る部分

AIで進めやすいもの:

- 構成案
- デザイン方向性の案出しと批評
- 実装
- 顧客情報を元にしたコピー草案
- 問い合わせ分類と返信文の下書き
- QAチェックリスト
- 提案書のたたき台

人が必ず見るべきもの:

- ライセンス判断
- 最終的なブランド判断
- ブランドの最終トーン
- 実在の顧客情報
- 予約確定、価格例外、ポリシー例外
- 公開前確認
- 価格確定

## 生成コマンド

```bash
ai-automation-kit website-side-hustle --industry hospitality --client-type local-business --niche tourism-hotel --output .tmp/website-side-hustle
```

この出力には、公開情報カタログ、顧客ヒアリング、Codex用ワークフロー、予約・問い合わせ運用テンプレート、提案書、価格表、保守案、QA、公開チェック、編集可能なサンプルサイト、そして裏側の簡易管理画面モックが入ります。

初心者が最初に見るべき生成ファイル:

- `ai_agent_handoff.md`: Codex、Claude Code、Cursor などに渡す共通指示
- `designer_grade_agent_playbook.md`: AIエージェントでデザイン品質を上げるための批評・修正ループ
- `public_ai_design_sources.md`: 公開AI/デザイン/OSS情報をどう使うかの整理
- `website_quality_gate.md`: Core Web Vitals、アクセシビリティ、SEO、フォーム、信頼性、人間承認を含む最終品質ゲート
- `homepage_review_scorecard.csv`: ホームページの納品準備度を点数化する表
- `agent_design_review_prompt.md`: AIエージェントにシニアレビュアー視点で確認させるプロンプト
- `client_kickoff_questions.md`: 最初の打ち合わせで何を聞くかを迷わないための質問表
- `beginner_human_guide.ja.md`: 人間が読むためのやさしい手順書
- `reservation_inquiry_system.md`: 問い合わせ・予約の裏側設計
- `delivery_acceptance_checklist.md`: 公開・納品前にお客様と確認するチェックリスト
- `client_handoff_note.md`: 納品後に毎日どう使うかを説明する引き継ぎメモ
- `inquiry_dashboard.html`: ブラウザーで見られる裏側の管理画面モック
