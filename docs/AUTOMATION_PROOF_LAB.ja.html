第 1 節これは何か、何ではないか

これは何か

  • 1つの業務パックを、ローカルで検証するための作業場所です。
  • 証拠、承認済み完成例、必須ゲート、過去ケースを整理するための道具です。
  • 人の実結果と、候補システムの想定結果を比較する場所です。
  • not_ready や assist_only も価値ある結論として返す道具です。

これは何ではないか

  • 外部システムを実行するエンジンではありません。
  • Human Approval Edition の承認境界を弱める道具ではありません。
  • どの業務でも必ず L3 や L4 へ進めるという約束ではありません。
  • 副業収入を保証する道具でもありません。

Proof Lab が行うこと

証拠を集める手順書、承認済み完成例、責任者メモ
→
監視運転で比較人の結果と候補システムの結果を照合
→
必須ゲートを判定復旧、二重防止、例外担当、データ利用許可など
→
準備度を返すL0-L4 と報告書、改善ロードマップ

ここで終わりです

人間が残す仕事

ルール確認AI は候補まで、人が正式化
→
承認境界の維持今ある毎回承認は消えない
→
納品判断共有、保留、改善、中止を決める
大事な原則。 点数が高くても、必須ゲートが落ちていれば昇格しません。復旧不足、例外担当不足、監視運転の証拠不足は、通常ケースの見栄えでは打ち消せません。

第 2 節インストール

まずキット全体を入れます。Proof Lab も同じローカル前提で動きます。

git clone https://github.com/goonobu-dot/ai-automation-starter-kit.git
cd ai-automation-starter-kit
python3 -m venv .venv
source .venv/bin/activate
python3 -m pip install --upgrade pip setuptools
python3 -m pip install -e .
ai-automation-kit --version
ai-automation-kit 0.1.0

始める前に環境診断を1回入れてください。

ai-automation-kit doctor --output .tmp/proof-lab-doctor
status=warning
report=.tmp/proof-lab-doctor/doctor_report.md

ready は問題なし、warning は注意しつつ続行可、blocked は先に対処が必要です。

第 3 節初期化と、正確なフォルダー構成

Proof Lab は、1つの業務パックごとに1つの作業場所を作ります。共有ドライブや本番フォルダーを直接指さず、ローカルに閉じて始めてください。

ai-automation-kit autopilot-proof-lab init \
  --pack-id invoice-order-check-daily \
  --organization "Example Co" \
  --objective "請求書照合を安全に検証する" \
  --workspace .tmp/proof-lab-invoice \
  --language ja
assessment_id=asmt_...
status=created
stage=created
decision=not_ready
maximum_level=L0
next_action=Add current-work evidence and accountable owners.
external_actions_enabled=false

init の後、Proof Lab が次の管理用構成を作ります。.proof_lab の中は手で書き換えず、下のコマンドで操作してください。

proof-lab-invoice/ ├── .proof_lab/ │ ├── state.json │ ├── pack_manifest.json │ ├── audit.jsonl │ ├── evidence/ │ │ └── ev-0001.md │ └── cases/ │ └── case-001/ │ ├── input.json │ ├── expected.json │ └── proposed.json └── 05_REPORTS/ ├── current_workflow.md ├── automation_scope.md ├── source_inventory.csv ├── decision_table.csv ├── exception_register.csv ├── connector_permissions.md ├── recovery_plan.md ├── kill_switch_plan.md ├── shadow_test_cases.csv ├── comparison_report.md ├── readiness_decision.json ├── readiness_report.html ├── improvement_roadmap.md └── proposal_scope.md
普通のファイルだけを置いてください。 シンボリックリンク、実行スクリプト、秘密情報の丸ごと書き出し、未確認の本番データは置かないでください。Proof Lab は証拠ハッシュとスナップショット前提なので、入口の整理が重要です。

第 4 節証拠とケースの正確な準備

証拠の準備

add-evidence の前に、元資料を管理用作業場所の外で役割ごとに分けます。コマンドを実行すると、選んだファイルが安全にコピーされ、ハッシュが記録されます。

元資料の整理例ここへ置くものなぜ必要か
source-evidence/current-process/現在手順のメモ、担当者チェックリスト、入力画面の説明、個人情報を抜いた CSV などいま実際にどう始まり、どう終わるかを示すため
source-evidence/approved-examples/人が承認した完成物「正しい結果」の基準にするため
source-evidence/policies/社内ルール、契約条件、金額上限、止める条件AI が勝手にルールを作らないため
source-evidence/exceptions/例外案件の記録、エスカレーションメモ例外を見つけて人へ戻せるか確かめるため
source-evidence/recovery/取消手順、支払保留手順、緊急停止手順、復旧確認メモ補助以上へ進むための必須材料だから

ケースの準備

過去ケースは、同じ仕事を3つの見方でそろえます。

  • 入力証拠: そのとき見えていた資料
  • 期待される人の結果: 実際に承認された人の完成結果
  • 候補システムの想定結果: 外部実行なしで「もし走ったらどうなるか」

監視運転では、少なくとも次の7種類を用意してください。

必須ケース

  • normal
  • near_limit
  • missing_data
  • duplicate

さらに必要

  • conflicting
  • expired
  • external_failure

11個の必須ゲート

各ゲートは pass、fail、unknown のどれかです。点数が高くても、不合格や不明を飛び越えることはできません。

ゲートID人が確認する意味
input_identifiable1件の仕事に必要な入力を区別できる。
completion_testable「正しく完了した」を同じ方法で検査できる。
standard_policy_confirmed通常時のルールを責任者が確認している。
exceptions_detectable異常案件を実行前に見つけられる。
exception_owner_assignedAIが決められない案件を受け取る人が決まっている。
least_privilege_available将来の実行環境を必要最小限の権限にできる。
idempotency_defined再試行や重複で同じ処理を二重実行しない。
recovery_defined保留、取消、再開の手順が書かれ、確認できる。
kill_switch_owned停止ボタンを扱う責任者が決まっている。
data_use_permittedその資料を検証へ使う許可がある。
shadow_test_passed過去ケース比較が合意した品質基準を満たした。
初心者向けの考え方。 10件しか集まっていなくても学びにはなります。ただし L3 や L4 の根拠には足りません。その場合は assist_only、not_ready、または「監視運転を続ける」を誠実な結論として納品します。

第 5 節操作の流れ

1

証拠を登録する。 各ファイルに役割、提供者、分類を付けます。こうしておくと、結論と証拠がきれいに結びつきます。

ai-automation-kit autopilot-proof-lab add-evidence \
  --workspace .tmp/proof-lab-invoice \
  --source ./source-files/approved-example-001.md \
  --role approved_example \
  --provided-by "Operations owner" \
  --classification confidential
evidence_id=ev-0001
status=present
external_actions_enabled=false
2

過去ケースを登録する。 実際に起きた1案件ごとに、入力、人の結果、候補システムの結果を束ねます。

ai-automation-kit autopilot-proof-lab add-case \
  --workspace .tmp/proof-lab-invoice \
  --case-id case-001 \
  --input ./cases-source/case-001/input.json \
  --expected ./cases-source/case-001/expected.json \
  --proposed ./cases-source/case-001/proposed.json \
  --case-class normal \
  --risk-tier low \
  --expected-route standard \
  --proposed-route standard
case_id=case-001
external_actions_enabled=false
3

必須ゲートを記録する。 合格、不合格、または不明を明記し、責任者と根拠を残します。

ai-automation-kit autopilot-proof-lab set-gate \
  --workspace .tmp/proof-lab-invoice \
  --gate recovery_defined \
  --status fail \
  --owner "Procurement systems owner" \
  --evidence-id ev-0001 \
  --note "No tested vendor deactivation path yet"
gate=recovery_defined
status=fail
external_actions_enabled=false
4

評価する。 比較、ゲート判定、準備度決定、報告書生成をまとめて行います。

ai-automation-kit autopilot-proof-lab evaluate \
  --workspace .tmp/proof-lab-invoice
status=not_ready
stage=not_ready
decision=not_ready
maximum_level=L0
next_action=Resolve the blocking input or completion definition before automation.
external_actions_enabled=false
5

状態を確認する。 変更後は必ず status を見ます。次に何をするか、判断が古くなっていないかが分かります。

ai-automation-kit autopilot-proof-lab status \
  --workspace .tmp/proof-lab-invoice
status=stale_decision
stage=stale_decision
decision=not_ready
maximum_level=L0
next_action=Restore changed evidence or reassess the changed configuration.
external_actions_enabled=false
6

ローカル画面を開く。 同じPCのブラウザで使います。公開 Web アプリではありません。

ai-automation-kit autopilot-proof-lab serve \
  --workspace .tmp/proof-lab-invoice \
  --language ja \
  --no-open
http://127.0.0.1:8080/

第 7 節L0-L4、監視運転、古い証拠、誠実な中止

L0自動化不可。必要証拠、ルール、責任者、復旧のどれかが不足しています。
L1AI補助。証拠整理や下書きはできるが、判断は人に残します。
L2監視運転。本番のつもりで判定するが、外部変更はせず、人の結果と比較します。
L3条件付き自動化の準備完了。将来の別実行系で、標準案件だけを進められる状態です。
L4無人標準経路の準備完了。将来の別実行系で、停止・隔離・復旧・監査つきで回せる状態です。

監視運転を平たく言うと

「もし本番だったらどう動くか」を試すけれど、外へは何もしない運転です。候補システムの振る舞いを、人が実際に承認した結果と照らし合わせます。

監視運転で見ること意味
完全一致・実質一致通常ケースで人の実務と合っているか
誤検知・見逃し危険な失敗が平均値に埋もれていないか
二重実行の扱い再実行しても事故らないか
例外検出おかしな案件を人へ戻せるか
復旧シミュレーション間違えたときの戻り道があるか

古い証拠（stale evidence）

次のような変更があれば、以前の判定は古くなります。

  • 証拠ファイルが編集、差し替え、削除された
  • ポリシー版が変わった
  • 責任者が変わった
  • 新しい例外経路が見つかった
  • ケースや候補ルールが前回評価後に変わった
古い判定は止める合図です。 古い判定を根拠に、試験導入や外部実行計画を進めてはいけません。status を見て、変更を確認し、必要なら evaluate をやり直してください。

誠実な中止

「できません」は価値のある納品です。Proof Lab は次を分けて伝えるべきです。

  • 一時的な不足: 証拠や整備がそろえば再挑戦できる
  • 費用対効果が悪い: やれても元が取れない
  • 責任上やめるべき: 法務、人事、金銭、契約などで人判断を残すべき
  • 技術的に無理: 安全な開始条件、入力品質、復旧経路が作れない

第 8 節副業納品、プライバシー、安全

副業としてどう納品するか

完全自動化を約束しなくても、この評価作業は売れます。初心者が渡しやすい納品一式は次です。

  • readiness_report.html：非技術者向けの説明用
  • readiness_decision.json：厳密な判定結果
  • comparison_report.md：監視運転の比較結果
  • improvement_roadmap.md：不足ごとの担当者、対応、合格条件
  • proposal_scope.md：何を人に残すか、その理由
売り込みすぎないこと。 結果が not_ready や assist_only なら、それを価値として説明してください。危ない約束を防いだこと自体が成果です。

プライバシーと安全のルール

  • できる限りマスキング済み、または縮小したサンプルを使います。
  • 秘密鍵、パスワード、生の本番トークンを置かないでください。
  • 支払データ、人事データ、権限付与データを公開デモ用フォルダーへ混ぜないでください。
  • ローカル画面は 127.0.0.1 のまま使ってください。
  • AI が提案した内容は、人が確認するまで正式ルールではありません。

第 9 節困ったとき

コマンドが見つからない

source .venv/bin/activate で仮想環境を有効化し直し、python3 -m pip install -e . を再実行してください。

判定が stale になった

ai-automation-kit autopilot-proof-lab status --workspace ... を実行し、何が変わったかを見ます。意図した変更なら再評価、心当たりがないなら作業を止めて中身を確認します。

ブラウザ画面が開かない

ai-automation-kit autopilot-proof-lab serve を実行した端末が動いたままかを確認し、表示された 127.0.0.1 のURLをそのまま開いてください。ポート競合なら起動し直して新しいURLを使います。

ずっと L1 や L2 のまま

まず必須ゲートの不合格を見ます。よくある原因は、復旧不足、二重防止不足、例外担当不足、監視運転の証拠不足です。対処は improvement_roadmap.md に出る前提です。

お客様が「今すぐ完全自動化して」と言う

約束しないでください。準備度レポートを見せ、現行の人間承認境界を説明し、外部実行は将来の別実行系と追加の法務・セキュリティ確認が必要だと伝えます。

安全な止まり方。 証拠より大きな約束をしそうになったら、assist_only、not_ready、または監視運転継続で止めてください。それがこのProof Labの役目です。
