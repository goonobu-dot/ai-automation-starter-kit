Phase 1A local workflow

月報オペレーター作業場所マニュアル

このマニュアルは、初心者が月報用のローカル作業場所を安全に回すための手順書です。インストール、Codex ログイン確認、Codex への作成依頼、3種類のファイル配置、確認、回答、下書き作成、停止とやり直し、承認、翌月準備までを順番に示します。

正直な公開範囲

この版は月報用の単一パックだけです。既存の flow project を置き換えるものではなく、ローカルの月次運用入口を追加した段階です。

対応OSの事実

作成と承認更新は macOS または Linux のみです。Windows では読むことはできますが、安全な更新経路はこの版では提供しません。

安全の事実

APIキーは不要です。外部送信はしません。承認はこの端末上のローカル記録です。

最初に開く資料

対応するセットアップ文書

Codex への最初の依頼文は START_WITH_CODEX.ja.md にあります。そこで参照するリポジトリ契約は AGENTS.md、全体索引は INDEX.md、英語版は office-workspace.html です。

このマニュアルは、作業場所をこれから初回運用する人向けです。すでに作業場所ができている場合は、フォルダ全体像と確認手順から読めば十分です。

インストールと Codex ログイン確認

最初にローカル Python 環境へキットを入れ、その後で Codex のローカルログイン状態を確認します。この流れはチャットへ資格情報を貼る方式ではありません。

リポジトリ上でキットを入れる

CLI と月報パックを、確認するローカルファイルと同じ端末に置きます。

python3 -m venv .venv
source .venv/bin/activate
python3 -m pip install --upgrade pip setuptools
python3 -m pip install -e .
作成前に Codex ログインを確認する

Codex 主導のセットアップ前に、ローカルのログイン状態が整っている必要があります。

codex login status
ログイン未完了ならここで止める

codex login status が失敗したら、人間がローカルのログインを完了させてから再開します。APIキー要求へ切り替えないでください。外部送信で回避できるようにも説明しないでください。

Codex へ作業場所作成を依頼

START_WITH_CODEX.ja.md の冒頭3行を、そのまま Codex に渡します。Codex は保存先、作業名、承認者、対象月の4点を1つずつ確認してから CLI を実行します。

正式な作成経路

Codex は ai-automation-kit office-workspace create を実行し、その直後に ai-automation-kit office-workspace status --workspace で状態確認します。

勝手な追加はしない

独自の実行ファイル指定、shell 回避、approval bypass、remote send、危険な shortcut flag は足さないでください。入口を狭くしているのは安全のためです。

ai-automation-kit office-workspace create \
  --root "./workspaces" \
  --name "Construction Monthly" \
  --approver "Operations Manager" \
  --pin "482913" \
  --period "2026-07" \
  --language ja

ai-automation-kit office-workspace status \
  --workspace "./workspaces/Construction_Monthly" \
  --json

検索しやすい1行版: ai-automation-kit office-workspace create --root ./workspaces --name "Construction Monthly" --approver "Operations Manager" --pin 482913 --period 2026-07 --language ja と ai-automation-kit office-workspace status --workspace ./workspaces/Construction_Monthly --json。

作業前にフォルダ全体像を見ておく

月報オペレーター作業場所は、誰が見ても分かるローカルフォルダ構成を前提にしています。各手順が具体的な置き場に結びつくので、初心者でも何をどこへ入れるかを先に把握できます。

キットをローカルへ入れ、codex login status を確認します。

Codex へ依頼

Codex が1問ずつ確認し、正式な CLI で作業場所を作成します。

3種類のファイルを正しいフォルダへ置き、今月資料は対象月フォルダだけにまとめます。

テキスト版: インストールとログイン確認を行い、Codex に月報作業場所作成を依頼し、過去完成月報と過去参考資料と今月資料を所定の場所へ置き、対象月を確認し、不足質問へ1つずつ答え、下書きを作り、必要なら停止してやり直し、ローカル承認を行い、最後に次月を準備します。

00_START_HERE

最初に開く概要です。分かりやすい説明と workspace status が置かれます。

05_DRAFTS

下書きはここに出ます。承認前の確認場所です。

06_APPROVED_OUTPUTS と 07_AUDIT

承認済みコピーと監査履歴がローカルに残ります。外部送信はしません。

3種類のファイル置き場

初回の確認前に、3種類のファイルの意味を理解しておくことが重要です。フォルダの役割が曖昧だと、下書きの根拠も曖昧になります。

フォルダ 入れるもの 具体例
01_APPROVED_PAST_OUTPUTS すでに承認済みの過去月報です。文体、構成、見出しの手本になります。 2026-05-monthly-report.md、2026-06-monthly-report.docx
02_PAST_SUPPORTING_FILES 過去の補助資料です。定例の指標名、前提条件、背景メモを補います。 vendor-summary.xlsx、policy-export.pdf、recurring-notes.md
03_CURRENT_INPUTS/<YYYY-MM> 今月の資料だけを入れます。対象月フォルダに閉じ込めるのが大切です。 2026-07/labor_hours.csv、2026-07/delivery_costs.xlsx、2026-07/site-meeting-notes.md
3種類を混ぜない

今月の生資料を 01_APPROVED_PAST_OUTPUTS に入れないでください。未承認の下書きを過去完成物の見本にも使わないでください。この分離自体が安全設計です。

対象月を確認する

ファイルを置いたら、まずローカルの確認を実行します。どのファイルが受理されたか、次が質問か下書き作成かをここで判断します。

ai-automation-kit office-workspace inspect \
  --workspace "./workspaces/Construction_Monthly" \
  --period "2026-07"

1行版の確認コマンド: ai-automation-kit office-workspace inspect --workspace ./workspaces/Construction_Monthly --period 2026-07。

  • 資料を差し替えたら、重要な変更後に再確認します。
  • 確認は外部送信せず、ローカルの source manifest と状態だけを更新します。
  • questioning なら次は回答です。無理に下書き作成へ進みません。

質問へ1つずつ答える

この作業場所は、不足している必須質問だけを出します。回答は短く、事実ベースで、今月の文脈に閉じたものにしてください。回答操作は localhost UI から行います。

よい回答

閲覧者は運営会議と経理責任者です。 のように、文体や説明の深さが決まる回答です。

避けたい回答

いい感じで のような曖昧な表現は、下書きの推測を増やし、修正回数を増やします。

回答が保存されると、現在の対象月は ready_for_run になります。その状態で下書き作成へ進みます。

下書きを作る

下書き作成もローカルです。Codex は隔離された実行場所で安全なスナップショットだけを読み、検証後に 05_DRAFTS へ昇格できるファイルだけを返します。

作成後に見えるもの

対象月はレビュー待ちとなり、05_DRAFTS/<YYYY-MM> に下書きが現れ、UI では根拠や未解決項目を確認できます。

フォルダを開いて、承認前のドラフトとして読みます。ここではまだ完成版扱いにしません。

1行版のUI起動コマンド: ai-automation-kit office-workspace serve --root ./workspaces --language ja --no-open。

停止してやり直す

誤った資料が入っていた、回答を直したい、実行を止めたいといった場合は、いったん停止してローカル状態を直してから同じ月で再実行します。承認済み成果物や受理済み入力は守られます。

停止が向く場面

  • 今月フォルダに別月のファイルが入っていた。
  • 回答内容が関係者レビューで変わった。
  • 実行が止まり、資料修正が必要な不足質問が出た。

次の動き

  • フォルダ内容または回答を直す。
  • 資料を変えたなら再確認する。
  • 同じ対象月で下書き作成をやり直す。

ローカル承認して保存する

承認すると、レビュー済みのローカル下書きが 06_APPROVED_OUTPUTS へコピーされ、07_AUDIT に監査記録が追加されます。承認者名と設定済み PIN が承認条件です。

05_DRAFTS

下書きを読み、元資料と照らし合わせる場所です。

06_APPROVED_OUTPUTS

承認済みのローカル完成版が保存され、固有の hash を持ちます。

07_AUDIT

監査履歴を残し、次月の文体参照を正直に辿れるようにします。

approved hash は重要です

承認済み hash があることで、次月は「どの完成版を文体参照にしたか」を曖昧にせず記録できます。監査履歴と一緒に残してください。

次の月を準備する

承認後は次月を作り、承認済み前月を文体参照にするかを明示的に決めます。次月準備は append-only で、前月データを消しません。

  • 前月はそのまま残ります。
  • 文体参照に使えるのは承認後の完成版だけです。
  • 次月入力フォルダは空から始まり、今月資料と混ざりません。

AIが判断できることと人間が判断すること

AIが判断できること
  • 受理済みのローカル資料スナップショットを読むこと。
  • 下書き文案、章立て、不足事項の指摘を行うこと。
  • ホストが認めた場合のみ承認済み文体参照を使うこと。
  • レビュー用のローカル下書きを準備すること。
人間が判断すること
  • どのファイルが使用許可済みか。
  • 読み手、数値、表現が正しいか。
  • 停止、再実行、承認、次月準備の判断。
  • ローカルレビュー後に外部へ提出するかどうか。
ローカルシステムが守ること
  • APIキーは不要です。
  • 外部送信はしません。
  • 原本入力は変更しません。
  • 承認済みコピーと監査記録をディスク上に残します。
ローカル限定の運用

このマニュアルは意図的に範囲を絞っています。メール送信、Slack 投稿、公開、支払い、運用DB更新は、この作業場所の対象外です。

トラブル対応

起きたこと よくある意味 次にやること
Codex ログイン確認が失敗した この端末のローカル Codex セッションが未準備です。 ローカルのログインを完了し、codex login status を再実行してから再開します。
Windows で作成が止まった Phase 1A の安全な作成と承認更新が未対応です。 macOS または Linux の端末で作成と承認更新を行います。
確認で有効なファイルが見つからない 今月フォルダが空か、置き場が間違っています。 01_APPROVED_PAST_OUTPUTS、02_PAST_SUPPORTING_FILES、03_CURRENT_INPUTS/<YYYY-MM> を見直します。
質問待ちのまま進まない 必須回答がまだ足りません。 localhost UI から回答し、必要なら再実行します。
下書きをそのまま信用できない まだ承認前で、レビュー待ちの状態です。 05_DRAFTS の下書きを元資料と照合し、確認後に承認します。
手動復旧が必要と表示された 状態ファイル、run、lock の確認が必要です。 00_START_HERE/workspace_status.json、.system/workspace.json、.system/periods/<YYYY-MM>/state.json、.system/runs/、07_AUDIT/audit.jsonl を見てから次の判断をします。
月次処理をいったん止めたいとき

受理済みファイルはそのままにし、不明点だけを記録して承認前で止めます。確証のない承認より、一度止める方が安全です。

次月で文体を引き継ぎたいとき

前月を先に承認し、その上で次月準備時に文体参照として確認します。未承認下書きを基準にしないでください。
