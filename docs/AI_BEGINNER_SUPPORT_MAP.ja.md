# AI初心者サポートマップ

この文書は、AI初心者がこのプロジェクトを使って業務自動化を提案・導入するときに、今の技術でAIが支援できることと、人間が必ず行うことを分けて整理したものです。API、フォルダ、クラウド設定、デプロイ、顧客対応にまだ慣れていない人を想定しています。

## まず、これだけやってください

1. 毎週くり返している小さな作業を1つ選ぶ。
2. このページをAIエージェントに読ませる。
3. AIに1問ずつ質問してもらう。
4. 架空データ、または伏せ字にしたサンプルデータを用意する。
5. 本物のAPIキー、password、顧客の個人情報、本番アクセス情報は貼らない。
6. 本番送信や本番データ変更の前に、dry-runで試す。

## やさしい言葉での説明

- APIキーとは、外部サービスを使うためのパスワードのような秘密の値です。チャットには貼りません。
- クラウドアカウントとは、AWS、Google Cloud、Azure、Render、Railway、Vercel、DigitalOcean、Fly.io などでシステムを動かすためのアカウントです。
- dry-runとは、本番送信や本番データ変更をせずに、下書き、確認キュー、レポート、ファイルだけを作る安全なテストです。
- 人間承認とは、メッセージ送信、業務データ変更、課金利用、クラウド本番化の前に、人が確認してOKを出すことです。
- 受付フォルダとは、AIが確認する前に、依頼やファイルを一旦集める場所です。

## 基本方針

AIは、案内、下書き、確認、分類、ファイル生成、エラー説明、安全なコマンド案の作成を支援できます。人間は、アカウント、課金、顧客アクセス、secret、本番通信、法的な境界、顧客への約束を承認します。

売る前、作る前、公開する前にこのマップを使ってください。

## 今の技術でAIが支援できること

| 領域 | AIが支援できること | 人間が必ず行うこと |
|---|---|---|
| 業務選定 | 候補フローを比較し、1問ずつ聞き出し、小さなPoCを勧める。 | 顧客に本当にその課題があるか確認する。 |
| 顧客ヒアリング | 質問を作り、曖昧な回答を不足リストにし、追加確認メールを作る。 | 顧客と話し、業務事実を確認する。 |
| 入力設計 | email、form、LINE、Slack、spreadsheet、CRM、webhook、local folder、storage bucketのどれか整理する。 | 実システムへのアクセスを許可する。 |
| サンプルデータ | 架空データを作り、実例を伏せ字にし、必要な列を定義する。 | 個人情報を消し、共有可能か承認する。 |
| APIキー | どこでキーを作るか、どの値が必要か説明する。 | アカウント作成、課金、キー発行、チャット外での保存を行う。 |
| 受付フォルダ | ファイルの到着場所と命名ルールを決める。 | 実際のフォルダを作り、権限を設定する。 |
| 出力設計 | approval queue、report、reply draft、spreadsheet row、ticket summary、local outboxを作る。 | 顧客や業務記録に影響するものを承認する。 |
| ローカルdry-run | 安全なローカルテスト計画とチェックリストを作る。 | 自分のPCで実行するか、実行者を承認する。 |
| クラウドアカウント | AWS、Google Cloud、Azure、Render、Railway、Vercel、DigitalOcean、Fly.ioを大まかに比較する。 | クラウドアカウント、課金、IAM、domain、本番設定を作る。 |
| トラブル対応 | secretを消したエラーを読み、原因候補と次の確認を出す。 | 本番設定を変えるか判断する。 |
| rollback | rollback手順と障害メモを作る。 | rollbackを承認し、関係者へ連絡する。 |
| 顧客提案 | 小さな有料dry-run PoCに変換する。 | 価格、契約、顧客期待値を決める。 |

## 人間が必ず行うこと

このプロジェクトは、AIがアカウント管理者や顧客の意思決定者を完全に置き換えるとは言いません。人間が必ず行うことは次です。

- 顧客に提案する業務を選ぶ、または承認する
- 自動化してよい業務か確認する
- クラウドやSaaSのアカウントを作る
- APIキーやOAuth appを作る
- secretを `.env`、secret manager、provider console などチャット外に保存する
- folder、inbox、sheet、CRM、webhookへのアクセスを承認する
- 顧客データを使ってよいか確認する
- 本番送信の前に承認する
- 課金を承認する
- go-liveを承認する
- rollbackを承認する
- 有料dry-run PoCの範囲を承認する

## 初心者の進め方

1. READMEを読む。
2. `docs/AI_AGENT_GRILL_ME_SKILL.ja.md` を開く。
3. スキル文をAIエージェントへ貼る。
4. AIに1問ずつ聞いてもらう。
5. 複数ではなく、1つの業務フローを選ぶ。
6. 入力元を決める: inbox、form、LINE、spreadsheet、folder、webhook、CRM。
7. 出力を決める: draft reply、report、checklist、approval queue、local outbox、ticket、spreadsheet row。
8. 人間の承認者を決める。
9. 架空または伏せ字のサンプルデータを用意する。
10. local dry-runを行う、または計画する。
11. ローカルで価値が見えた後に `cloud-plan` を使う。
12. 範囲、承認、停止条件が明確になったら有料dry-run PoCを提案する。

## 自動化前に必要な最小入力

どの自動化でも、実装前に次を答えます。

- 業務課題: 何が繰り返し発生しているか。
- 担当者: 今は誰が対応しているか。
- 入力: 依頼やファイルはどこに届くか。
- 出力: 何を作るか。
- 承認者: 誰が確認するか。
- 件数: 週または月に何件あるか。
- ツール: どのアプリやサービスが関係するか。
- データ境界: AIへ送ってはいけないものは何か。
- 本番境界: 承認なしに動かしてはいけないものは何か。
- 成功指標: PoCを続ける価値は何で判断するか。

## 最初に向いている案件

最初に向いているのは、小さく、見えやすい案件です。

- 請求書フォロー下書き
- 不足書類リマインド
- 問い合わせ返信下書き
- 日次問い合わせ要約
- 予約前確認リスト
- FAQ振り分け
- 見込み客リサーチメモ
- 週次KPIレポート
- 経費ルール確認

最初から、重要判断、規制対象の助言、大規模移行、本番顧客送信、決済操作、取り消せないDB更新が必要な案件は避けます。

## 停止条件

次の場合は止めて設計を見直します。

- 承認者がいない
- 顧客がサンプルデータを出せない
- 法務、医療、金融、HR判断が必要
- dry-runの証拠なしに本番送信が必要
- クラウドアカウントの所有者が不明
- 本物のsecretをチャットへ貼ろうとしている
- 提案が収益保証や完全自動化を約束している

## AIに貼る依頼文

```text
AI初心者サポートマップを読んで、私に1問ずつ質問してください。
業務フロー、入力元、出力、人間承認ゲート、不足しているアカウント、APIキー、受付フォルダ、クラウドアカウント、rollback計画、有料dry-run PoCの範囲を整理してください。
本物のsecretはチャットで求めないでください。
```
