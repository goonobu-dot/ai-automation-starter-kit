# 最初の顧客案件ウォークスルー

このページは、AI初心者がこのプロジェクトを使って、最初の企業向け業務自動化案件を安全に進めるための実践手順です。

目的は、いきなり大きな自動化システムを売ることではありません。まず、企業の困りごとを1つ選び、サンプルデータで dry-run を作り、価値がありそうか一緒に確認することです。

## このページで目指す状態

最初の案件で目指すのは、次の状態です。

- 顧客の困りごとを1つ説明できる
- 入力、出力、承認者が分かっている
- サンプルデータで見本を作れる
- 本番送信しない dry-run として説明できる
- 顧客に見せるデモとレポートがある
- 有料PoCにする場合の範囲を説明できる

最初からクラウド、API、決済、本番連携まで全部やろうとしないでください。まずは「これなら役に立ちそう」と顧客が理解できる小さな証拠を作ります。

## Step 1: 売る相手を決める

最初は、次のような相手が向いています。

- 問い合わせ対応が多い店舗や教室
- 請求書や書類の確認が多い小規模事業者
- Google Sheetsで顧客管理している会社
- 毎週レポートを手作業で作っているチーム
- メールやLINEの返信が属人化している事業者

避けた方がよい相手:

- いきなり大企業全体へ本番導入したい相手
- 医療、法律、金融判断を自動化したい相手
- 承認者や責任者が決まっていない相手
- サンプルデータを出せない相手

## Step 2: 最初の商品を決める

初心者が最初に提案しやすい商品名は、次のような形です。

```text
業務自動化ミニ診断 + dry-run PoC
```

説明文:

```text
現在の繰り返し業務を1つ選び、サンプルデータでAI自動化の見本を作ります。
本番送信や本番更新はせず、下書き、作業キュー、承認リスト、レポートを作って、導入する価値があるか一緒に確認します。
```

この言い方なら、初心者でも無理な約束をせずに提案できます。

## Step 3: 顧客に聞くこと

最初の打ち合わせでは、技術の話をしすぎないでください。業務の困りごとを聞きます。

質問例:

1. 毎週または毎月、何度も発生している作業は何ですか。
2. その作業は今、誰が対応していますか。
3. 1件あたり何分くらいかかりますか。
4. 依頼や書類はどこに届きますか。
5. 最終的に何を作れば完了ですか。
6. AIが下書きした場合、誰が確認しますか。
7. サンプルデータを出せますか。
8. 本番送信や本番更新をしない見本なら試せますか。

顧客が答えられない場合は、案件を急がず、業務整理から始めます。

## Step 4: AIに質問係を任せる

AIエージェントに次の文章を貼ってください。

```text
私はAI初心者ですが、企業向け業務自動化の最初の案件を進めたいです。
この GitHub プロジェクト ai-automation-starter-kit の docs/FIRST_CLIENT_WALKTHROUGH.ja.md と docs/USER_MANUAL.ja.md を読んでください。

顧客へ提案する前に、私に1問ずつ質問してください。
質問は、業務課題、入力元、出力、承認者、サンプルデータ、使うツール、クラウドの必要性、有料PoCの範囲の順番にしてください。

本物のAPIキー、password、secret、顧客の機密情報はチャットに書かせないでください。
私が本番化を急いだ場合は、まず dry-run と人間承認を勧めてください。
```

## Step 5: 見本フォルダを作る

最初は請求書や書類フォローの例が説明しやすいです。

```bash
ai-automation-kit complete-workspace \
  --flow-id invoice-document-followup \
  --client-type local-business \
  --niche accounting \
  --output .tmp/first-client
```

まず見るファイル:

- `.tmp/first-client/FINAL_DELIVERY_GUIDE.md`
- `.tmp/first-client/client_command_center.html`
- `.tmp/first-client/demo_site/index.html`
- `.tmp/first-client/client_report/client_report.html`
- `.tmp/first-client/business_launch/START_HERE_BUSINESS_LAUNCH.md`

顧客に見せる時は、コマンドではなく、ブラウザで見られるHTMLやレポートを見せる方が伝わりやすいです。

## Step 6: 顧客用デモの話し方

デモでは、次の順番で話します。

1. 今の手作業を確認する。
2. サンプルデータだけで見本を作ったと説明する。
3. AIが作った作業キューを見せる。
4. AIが作った下書きを見せる。
5. 人間が確認する承認リストを見せる。
6. 本番化前に必要な設定を説明する。
7. 続けるなら有料PoCの範囲を決める。

言い方の例:

```text
これは本番送信ではありません。
まず業務の流れを見える化するための dry-run です。
AIが下書きや作業候補を作り、人間が確認してから次に進む形にしています。
```

## Step 7: 有料PoCの範囲を決める

最初の有料PoCは、範囲を小さくします。

含めるもの:

- 1つの業務フロー
- サンプルデータまたは伏せ字データ
- dry-run
- 作業キュー
- 下書き
- 承認リスト
- 顧客向けレポート
- クラウド導入に必要な情報の整理

含めないもの:

- 本番送信
- 本番DB更新
- 24時間無人運用
- 収益保証
- 法務、医療、金融判断の自動化
- 顧客アカウントの管理代行

## Step 8: クラウドへ進む判断

クラウドへ進むのは、次が揃ってからです。

- 顧客が価値を理解している
- 承認者が決まっている
- サンプルデータで dry-run ができた
- 使う connector が分かっている
- 課金とアカウント所有者が決まっている
- 停止手順と rollback が説明できる

進める場合:

```bash
ai-automation-kit guided-setup \
  --flow-id invoice-document-followup \
  --mode beginner \
  --deployment undecided \
  --connectors gmail,google-sheets \
  --output .tmp/first-client-guided-setup
```

```bash
ai-automation-kit cloud-plan \
  --flow-id invoice-document-followup \
  --provider aws \
  --workload scheduled-job \
  --connectors gmail,google-sheets \
  --output .tmp/first-client-cloud-plan
```

## 最初の案件での成功基準

最初の案件は、次ができれば十分に前進です。

- 顧客が「何が自動化できそうか」を理解した
- AIが作る下書きや作業キューを見せられた
- 本番化前に必要な情報が分かった
- リスクと承認ポイントを説明できた
- 有料PoCとして続けるか判断できた

初心者が信頼を得るには、できないことを隠さず、できる範囲を分かりやすく区切ることが大切です。このプロジェクトは、その説明と準備をAIに手伝わせるための道具です。
