AI Automation Starter Kit

統制ワークフロー6パック 初心者向け運用手順書

日々の照合、規程変更、品質事故、取引先登録、アクセス棚卸、助成金資料を、ローカルの確認用下書きにまとめるための手順書です。インストール、ログイン、正確なフォルダ、画面起動、資料確認、AIへの質問、下書きファイル名、PIN承認、翌日再利用、困ったときの対処、安全範囲、副業としての提案方法までを扱います。

対象業務は、表計算・台帳照合、規程変更の影響確認、品質事故・是正措置、取引先登録、アクセス権確認、助成金・補助金資料の6種類です。

1. この手順書でできること

最初に確認してください。 この6パックは、ローカルの確認用下書きを作るためのものです。統制承認、会計仕訳、助成金提出、アクセス変更、取引先有効化、是正予防措置の完了、メール送信、外部連絡は行いません。

対象は、毎日似た形の資料を集め、差分や不足を確認し、責任者が読むための下書きを作る仕事です。完全自動化ではありません。価値は、コピペ、初期確認、抜け漏れ整理、下書き作成を軽くしつつ、最終判断と外部アクションを人間に残すことです。

向いている現場毎日または頻繁に資料が発生し、承認者が決まっており、現在も人間が確認している業務です。
向いていない現場AIに統制証明、支払い、権限変更、法務判断、助成金の適格性証明を任せたい現場です。
基本の流れ資料を置く、読み取り結果を確認する、質問に答える、下書きを作る、資料と照合する、PINでローカル承認する、翌日に再利用する、という流れです。

メール自動化は対象外です。Gmail、Outlook、Slack、Teams、取引先への通知、申請先への連絡、顧客連絡などの外部送信は別プロジェクトとして扱います。この手順書は、あえてローカル確認と人間承認だけに範囲を絞っています。

2. 1日の見える流れ

1. パック選択6つの統制パックから1つ選び、最初の日付を YYYY-MM-DD で入力します。
2. 資料配置決められたフォルダに必要資料を置きます。当日資料と過去資料を混ぜません。
3. 資料確認受理、除外、拒否、足りない資料を、下書き前に確認します。
4. 質問回答重要度、エスカレーション、保留条件など、判断ルールを入力します。
5. 下書き作成05_DRAFTS に根拠付きの下書きファイルを作ります。
6. 人間確認数字、日付、担当者、結論を元資料と見比べます。
7. PIN承認承認者名とPINで、確認済みファイルを 06_APPROVED_OUTPUTS に保存します。

5. 作業場所を作る

初心者はローカル画面から始めるのが簡単です。パック一覧と日付形式が画面に出ます。

ai-automation-kit office-workspace serve --root "$HOME/AI-Office-Workspaces" --language ja

ターミナルに http://127.0.0.1:8080/ のようなアドレスが表示されます。同じパソコンのブラウザで開きます。この画面は公開用のウェブアプリではありません。127.0.0.1 のローカル画面として使います。

コマンドで作る場合の例です。2026年7月13日の表計算照合作業場所を作ります。

ai-automation-kit office-workspace create \
  --root ./workspaces \
  --name "Client A Spreadsheet Reconciliation" \
  --approver "Finance Manager" \
  --pin 482913 \
  --period 2026-07-13 \
  --pack spreadsheet-reconciliation-daily \
  --language ja

ai-automation-kit office-workspace status \
  --workspace ./workspaces/Client_A_Spreadsheet_Reconciliation \
  --json

PINはローカル承認のための番号です。銀行PIN、端末PIN、アカウントパスワードではありません。顧客のルールに合わせて、作業用の一時PINとして管理してください。

6. 正確なフォルダと入れるもの

00_START_HERE最初に状態メモを読みます。作業場所の現在地を知るための入口です。
01_APPROVED_PAST_OUTPUTS過去に承認済みの完成物を入れます。文体と構成の参考であり、今日の事実ではありません。
02_PAST_SUPPORTING_FILES規程、権限表、テンプレート、承認済みの統制文言、過去メモ、定義を入れます。
03_CURRENT_INPUTS/YYYY-MM-DD今日の根拠資料だけを入れます。2026年7月13日なら 03_CURRENT_INPUTS/2026-07-13 です。
04_QUESTIONS/YYYY-MM-DD情報が足りないとき、追加質問のファイルがここに出ます。
05_DRAFTS/YYYY-MM-DD下書きがここに出ます。承認済み成果物ではありません。
06_APPROVED_OUTPUTS/YYYY-MM-DD人間が確認し、承認者名とPINで承認したファイルだけがここに保存されます。
07_AUDIT承認記録と監査用のローカル記録があります。通常作業では手で編集しません。

パック別の入力例

spreadsheet-reconciliation-daily

ledger_export_2026-07-13.csv、bank_statement_2026-07-13.xlsx、任意の reconciliation_policy.md などを使います。下書きは、照合済み合計、未照合項目、不明担当者、締め時刻の問題、エスカレーション対象を整理します。仕訳登録、残高修正、支払い、照合サインオフは行いません。

policy-change-impact-daily

policy_notice_vendor_security.pdf、control_inventory.xlsx、任意の exception_log.csv などを使います。下書きは、影響を受ける統制、業務、教育、承認、未確認事項を整理します。規程公開、統制変更、教育割り当て、証跡依頼は行いません。

quality-incident-capa-daily

quality_incident_log.csv、capa_tracker.xlsx、任意の監査メモを使います。下書きは、重大度、封じ込め担当者、期限、検証状態、期限超過、根拠不足を整理します。是正予防措置の完了、製品判定変更、出荷承認、サプライヤー通知は行いません。

vendor-onboarding-daily

vendor_intake_register.xlsx、デューデリジェンス資料、税務メモ、銀行設定メモを使います。下書きは、登録可能、保留、不足資料ありを分けます。取引先有効化、税務書類受理、銀行情報更新、支払い、取引先連絡は行いません。

access-review-daily

access_review_export.csv、role_entitlement_matrix.xlsx、任意の入社異動退職ログを使います。下書きは、特権アクセス、ロール不一致、承認証跡不足、例外理由を整理します。権限付与、権限剥奪、棚卸認証、利用者連絡は行いません。

grant-packet-daily

grant_notice.pdf、eligibility_budget_workbook.xlsx、添付資料を使います。下書きは、適格性根拠、予算根拠、本文根拠、添付状況、期限、未解決質問を整理します。提出、適格性証明、署名、採択約束は行いません。

7. 実際の操作

先に資料を確認する

資料を置いたら、画面で「資料確認」を実行します。コマンドで行う場合は次の形です。

ai-automation-kit office-workspace inspect \
  --workspace ./workspaces/Client_A_Spreadsheet_Reconciliation \
  --period 2026-07-13

受理、拒否、除外されたファイルを確認します。期待した資料が読み込まれているか、必須資料が足りているかを見ます。必須入力が足りない場合は、無理に下書きを作らず、資料を追加するか、根拠がないことを明示します。

AIへの質問に答える

AIへの質問は、下書きの判断基準そのものです。短く、具体的に、責任者が確認できる形で答えます。

  • 照合: 「500ドル超の未照合、担当者不明の手数料、当日締め時刻の不一致はエスカレーション」。
  • 規程影響: 「顧客向け統制、規制対象業務、教育義務、役員承認に影響するものを重要影響とする」。
  • 品質事故: 「出荷済み製品の封じ込め未完了、検証期限超過、再発欠陥、根本原因担当者不明を重大リスクとする」。
  • 取引先登録: 「正式名称、税務書類、制裁確認、支払条件、担当承認、銀行根拠がそろうこと」。
  • アクセス棚卸: 「管理者権限、本番書き込み、財務承認、セキュリティ管理画面、共有所有者アカウントを特権範囲とする」。
  • 助成金資料: 「適格性は公募要領、申請者記録、予算表、添付根拠で確認できる場合だけ記載。不明は未解決にする」。

下書きを作り、元資料と見比べる

実行可能になったら下書きを作ります。下書きファイル名は正確に、spreadsheet_reconciliation_daily.md、policy_change_impact_daily.md、quality_incident_capa_daily.md、vendor_onboarding_daily.md、access_review_daily.md、grant_packet_daily.md です。

05_DRAFTS/YYYY-MM-DD のファイルを開き、数字、日付、担当者、システム名、取引先名、規程名、制度名、期限、金額、推奨事項を元資料と照合します。根拠がない文章は削除するか、「根拠不足」として残します。

確認後だけPIN承認する

承認すると、確認済み下書きが 06_APPROVED_OUTPUTS/YYYY-MM-DD にコピーされ、監査記録が残ります。承認しても、送信、公開、提出、本番更新、通知は行われません。外部で使うかどうかは、責任者が別途判断します。

8. 翌日に再利用する

承認後、次の日付を準備します。日次パックなので、2026年7月13日の次は 2026-07-14 です。画面に表示される場合、直近の承認済み成果物を文体と構成の参考として使えます。

  1. 今日の承認済みファイルが、翌日の参考にしたい内容か確認します。
  2. 画面から次の期間を準備します。
  3. 新しい日の資料だけを 03_CURRENT_INPUTS/2026-07-14 に置きます。
  4. 規程、表、テンプレートなどの固定資料は 02_PAST_SUPPORTING_FILES に残します。
  5. もう一度、資料確認を行います。昨日の事実を今日の事実として扱わせません。
  6. ルール、承認者、保留条件が変わった場合は、新しい質問に答えます。
  7. 下書き、資料照合、PIN承認を繰り返します。
再利用は学習ではありません。 過去の承認済みファイルを、構成や文体の確認済み参考として渡すだけです。AIモデルを恒久的に学習させる機能ではありません。

9. 統制業務の安全ルール

資料照合を省略しない見た目が整った下書きでも間違うことがあります。読み込まれた資料一覧と原本を確認します。
高リスク業務を自己承認しない財務、法務、品質、調達、セキュリティ、IT、助成金の責任者を承認者にします。
外部アクションに変えないメール自動化、アクセス変更、取引先有効化、支払い、助成金提出、規程公開はこの6パックの範囲外です。
不確実性を隠さない根拠がない場合は、不明または未解決として書きます。適格性、承認、根本原因をAIに推測させません。

実会社や顧客の資料を使う前に、必ずサンプルまたは匿名化資料で試してください。個人情報、財務情報、セキュリティ情報、医療、法務、輸出管理、助成金証明を扱う場合、その組織の責任者がAI利用可否を判断する必要があります。

10. 困ったとき

画面が開かない

office-workspace serve を実行したターミナルが動いているか確認します。表示された 127.0.0.1 のアドレスをそのまま開きます。ポートが使われている場合は、サーバーを起動し直し、新しく表示されたアドレスを使います。

Codexにログインできていない

codex login status を実行します。ローカルログインを完了してください。APIキー、パスワード、トークンをチャットや画面に貼り付けないでください。

日付が拒否される

YYYY-MM-DD を使います。例は 2026-07-13、2026-12-01 です。月だけ、存在しない日付、あいまいな表現は使えません。

ファイルが除外または拒否される

拡張子、破損、サイズ、ショートカット、クラウドのプレースホルダーではないかを確認します。パックごとに、.csv、.xlsx、.json、.pdf、.docx、.md、.txt の一部が使えます。

AIの質問が多すぎる

資料だけでは根拠が足りない可能性があります。規程、表、担当者一覧、期限、しきい値、過去の承認済み成果物を追加します。組織の正式ルールがある場合は、推測で答えないでください。

下書きが根拠なく断定している

その文章を削除する、根拠資料を追加する、または未解決として明示します。承認は、根拠と照合したうえで行います。

承認できない

承認者名、PIN、対象日、現在の下書きファイル名を確認します。回避策として 06_APPROVED_OUTPUTS に手でコピーしないでください。

手動復旧が必要と表示される

00_START_HERE/workspace_status.json、.system/workspace.json、.system/periods/YYYY-MM-DD/state.json、.system/runs/、07_AUDIT/audit.jsonl を確認します。手で直す前に、分かっている状態を整理して止まってください。

11. 副業としての提案方法

「コンプライアンスを完全自動化します」と言うのは危険です。初心者が売りやすい表現は、「1つの日次統制下書きワークスペースをローカルで作り、サンプルデータで効果を確認し、最終承認は責任者に残します」です。フォルダ、資料一覧、質問、下書き、承認記録、禁止事項が見えるため、相手も判断しやすくなります。

最初の有料検証の進め方

  1. 買い手を1人に絞ります。例は、財務責任者、コンプライアンス担当、品質責任者、調達責任者、ITまたはセキュリティ責任者、助成金担当者です。
  2. 毎日つらい確認作業を1つだけ聞きます。資料と承認者がある業務を選びます。
  3. デモは匿名化またはサンプル資料で行い、本番システムには触りません。
  4. 下書き時間、根拠不足の数、修正数、最終確認時間を測ります。
  5. 作業場所、この手順書、1ページの範囲表、人間だけが行う作業一覧を納品します。
  6. 恒久的な自律エージェントではなく、セットアップと1〜2回の確認サイクルとして価格を決めます。
完全自動化を約束しないでください。 正確性保証、統制承認、監査証明、アクセス変更、取引先有効化、助成金提出、支払い、外部連絡を自動で行うとは言わないでください。価値は、管理された下書き作成と再現できる確認手順です。
