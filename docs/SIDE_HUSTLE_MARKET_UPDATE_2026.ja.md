# 副業向けGitHub市場アップデート 2026

調査日: 2026-07-09

この資料は、GitHubで公開されている業務フロー自動化・AIエージェント・ブラウザ自動化・MCP関連プロジェクトの動きを、副業で中小企業に提案しやすい形へ変換したものです。

結論から言うと、今の副業で売るべきものは「AIで何でも自動化します」ではありません。

売るべきものは、次のような小さく安全な導入パックです。

> 1つの業務を選ぶ → サンプルデータでdry-run PoCを作る → AIが下書きする → 人間承認で止める → 効果を見せる → 継続導入を提案する

このキットの方向性は、最新のGitHub上の強いプロジェクトの流れと合っています。特に「人間承認」「dry-run」「ブラウザで見えるデモ」「MCP/コネクタ」「小さな有料PoC」は、今後も副業提案の中心に置く価値があります。

## 最新の流れから見えること

### 1. 企業は「AIチャット」より「業務に組み込むAI」を求めている

Dify、n8n、Activepiecesのようなプロジェクトが伸びている理由は、AIを単独のチャットとして使うだけでは業務改善になりにくいからです。

企業が本当に欲しいのは、次の形です。

- 問い合わせが来る
- 必要情報を整理する
- AIが返信文や確認リストを作る
- 人間が確認する
- 送信や更新は人間が判断する
- 結果を記録する

副業で提案するときも、「AIを導入しましょう」ではなく、「毎週発生しているこの作業を、下書き化して確認だけに近づけましょう」と伝える方が刺さります。

### 2. Human in the Loopは売り文句になる

activepieces/activepieces や Taskade系の流れを見ると、人間が途中で承認・上書き・確認できる設計が前面に出ています。

これは初心者向け副業では非常に重要です。

「AIが勝手に送信します」ではなく、次のように言うべきです。

> AIは下書きを作るだけです。実際の送信、予約確定、返金、価格変更、本番データ更新は、人間が確認してから行います。

この一言で、お客様の不安がかなり下がります。

### 3. ブラウザ自動化は魅力的だが、初心者の初手には危険

browser-use/browser-use や Skyvern-AI/skyvern の伸びを見ると、Webサイトや管理画面をAIが操作する領域は明らかに注目されています。

ただし、副業初心者が最初から「本番ブラウザをAIが操作します」と売るのは危険です。

最初に売るべきものは次です。

- 既存のWeb管理画面の作業手順を整理する
- スクリーンショットやサンプル入力で業務フローを見える化する
- AIに下書きや入力候補を作らせる
- 人間が画面へ転記する
- その後、十分に安全なら自動化候補を検討する

つまり、ブラウザ自動化は「最初の商品」ではなく「2段階目以降の商品」です。

### 4. MCPとコネクタは、実装より先に「接続設計」を売れる

modelcontextprotocol/servers や github/github-mcp-server の伸びは、AIが外部データやツールにつながる流れを示しています。

副業では、いきなりMCPサーバーを作る必要はありません。

むしろ最初に売れるのは、次の整理です。

- どのデータを見る必要があるか
- どのデータは見てはいけないか
- 誰がAPIキーを管理するか
- 誰が送信や更新を承認するか
- ローカルで試すのか、クラウドで動かすのか
- 失敗したらどこで止めるか

この「接続前の設計」は、AI初心者にも提供しやすく、顧客にとっても価値が伝わりやすいです。

## 売るべきもの

今の市場を踏まえると、このキットを使う人が副業で売るべきものは次の5つです。

## 5つの商品メニュー

### 1. 業務自動化ミニ診断

価格目安: 0円から3万円

内容:

- 30分から60分のヒアリング
- 繰り返し作業を1つに絞る
- 月の件数、1件あたり時間、承認者を確認する
- 自動化しない方がよい作業も明確にする

成果物:

- 業務候補リスト
- 最初に試す1業務
- やらないことリスト

提案トーク:

> まず、AIで全部自動化する話ではなく、毎週発生している1つの作業を見える化します。効果が出そうなものだけ、小さく試します。

### 2. dry-run PoC作成

価格目安: 5万円から15万円

内容:

- サンプルデータを使う
- AIの下書き出力を作る
- 承認待ちリストを作る
- 顧客に見せるHTMLデモやレポートを作る

成果物:

- `client_command_center.html`
- `client_report.md`
- `approval_queue.csv`
- `proposal_one_pager.md`

提案トーク:

> 本番送信はしません。まずサンプルデータで、どれくらい手作業が減りそうかを見える形にします。

### 3. 人間承認つき運用設計

価格目安: 5万円から20万円

内容:

- AIがやってよいことを決める
- 人間承認が必要な箇所を決める
- 例外時の止め方を決める
- 送信前チェックリストを作る

成果物:

- 承認フロー
- リスク境界表
- 例外対応ルール
- 運用チェックリスト

提案トーク:

> AIに任せるほど危ない部分ほど、人間承認を明確にします。自動化ではなく、まず安全に下書き化します。

### 4. コネクタ・クラウド準備支援

価格目安: 10万円から30万円

内容:

- Gmail、Google Sheets、LINE、Slack、Notionなどの接続要件を整理する
- APIキーや権限の管理者を決める
- ローカル実行かクラウド実行かを比較する
- 初期費用と月額費用の上限を決める

成果物:

- `guided_setup_questions.md`
- `cloud_provider_matrix.md`
- `secrets_and_env.md`
- `human_approval_required.md`

提案トーク:

> いきなりクラウドに載せるのではなく、必要な入力、権限、費用、停止方法を先に整理します。

### 5. ブラウザ作業の手順化と半自動化準備

価格目安: 10万円から30万円

内容:

- 管理画面で人がやっている作業を記録する
- 入力項目、判断条件、例外を整理する
- AIが作れる下書きや入力候補を決める
- 本番自動操作に進む前の安全条件を決める

成果物:

- 作業手順書
- 画面操作チェックリスト
- 入力候補生成ルール
- 本番自動化に進む条件

提案トーク:

> ブラウザ操作の完全自動化は、画面変更で壊れることがあります。まず人が確認しながら使える半自動化から始めます。

## 売らないもの

次は、初心者の副業では売らない方が安全です。

- 「完全自動化します」
- 「AIが勝手に顧客へ送信します」
- 「売上が必ず上がります」
- 「APIキーを送ってください」
- 「本番データをそのままAIに読ませます」
- 「ブラウザ操作を全部AIに任せます」
- 「契約書、医療、法律、人事判断をAIだけで処理します」

このキットの強みは、危険な自動化を急がず、dry-run PoCと人間承認で価値を見せることです。

## このキットに反映する判断

今回のGitHub市場確認から、このキットでは次の方針を強めます。

1. READMEと入口資料では、最初の商品を「小さな有料dry-run PoC」として説明する。
2. 副業資料では、n8nやDifyのような大規模ツールを直接売るのではなく、その前段階の「業務整理・デモ・承認設計」を売ると説明する。
3. ブラウザ自動化は魅力的だが、初心者向けには「手順化・半自動化準備・人間転記」から始めると明記する。
4. MCPやコネクタは、実装より前に「何を接続するか、誰が承認するか、どこで止めるか」を整理する商品にする。
5. 収益保証ではなく、作業時間削減の仮説、dry-run結果、顧客承認を販売の根拠にする。
