AI AUTOMATION STARTER KIT — AUTOMATION PROOF LAB MANUAL / v0.1.0

Automation Proof Lab Manual

This beginner manual explains the local proof lab for testing whether a workflow is honestly ready for stronger automation. It is a preparation and evaluation tool, not a switch that turns on external actions.

Japanese: 日本語

Local evidence onlyHuman approval stays in placeNo external actions

The proof lab does not send messages, pay vendors, update SaaS records, file forms, change permissions, or remove existing human approval. L3 and L4 are readiness labels for a future, separate runtime. No income is guaranteed.

SECTION 1What this is, and what it is not

What it is

  • A local proof lab for one workflow pack at a time.
  • A structured place to collect evidence, approved examples, hard-gate decisions, and historical cases.
  • A way to compare real human outcomes with proposed system outcomes before any future runtime is considered.
  • A safe route to useful conclusions such as not_ready or assist_only.

What it is not

  • Not an external action engine.
  • Not a way to weaken the Human Approval Edition.
  • Not a promise that every workflow can reach L3 or L4.
  • Not a business-income guarantee, even if a client pays for the assessment work.

What the proof lab does

Collect evidencePolicies, approved examples, and owner notes
→
Run shadow comparisonsExpected human result versus proposed system result
→
Evaluate hard gatesRecovery, idempotency, exception ownership, data use, and more
→
Return a readiness labelL0-L4 plus a report and roadmap

The proof lab ends here

What still requires humans

Confirm rulesAI may suggest; humans authorize
→
Keep approval boundariesCurrent pack approvals do not disappear
→
Decide deliveryShare the report, remain assisted, remediate, or stop
Important boundary. A good score never overrides a failed hard gate. Missing recovery, missing exception ownership, or failed shadow evidence blocks promotion even when routine cases look strong.

SECTION 2Install

Install the kit first. The proof lab uses the same local setup style as the rest of the project.

git clone https://github.com/goonobu-dot/ai-automation-starter-kit.git
cd ai-automation-starter-kit
python3 -m venv .venv
source .venv/bin/activate
python3 -m pip install --upgrade pip setuptools
python3 -m pip install -e .
ai-automation-kit --version
ai-automation-kit 0.1.0

Run a quick environment check before you start the proof lab:

ai-automation-kit doctor --output .tmp/proof-lab-doctor
status=warning
report=.tmp/proof-lab-doctor/doctor_report.md

ready means the local environment is fine. warning means you can usually continue, but you should read the report. blocked means fix the local setup first.

SECTION 3Init and the exact workspace folders

Start one proof-lab workspace per workflow pack. Keep the workspace local. Do not point it at a network share or a live production folder.

ai-automation-kit autopilot-proof-lab init \
  --pack-id invoice-order-check-daily \
  --organization "Example Co" \
  --objective "Test invoice order checks safely" \
  --workspace .tmp/proof-lab-invoice \
  --language en
assessment_id=asmt_...
status=created
stage=created
decision=not_ready
maximum_level=L0
next_action=Add current-work evidence and accountable owners.
external_actions_enabled=false

After init, Proof Lab creates this managed structure. Do not edit files inside .proof_lab by hand; use the commands below.

proof-lab-invoice/ ├── .proof_lab/ │ ├── state.json │ ├── pack_manifest.json │ ├── audit.jsonl │ ├── evidence/ │ │ └── ev-0001.md │ └── cases/ │ └── case-001/ │ ├── input.json │ ├── expected.json │ └── proposed.json └── 05_REPORTS/ ├── current_workflow.md ├── automation_scope.md ├── source_inventory.csv ├── decision_table.csv ├── exception_register.csv ├── connector_permissions.md ├── recovery_plan.md ├── kill_switch_plan.md ├── shadow_test_cases.csv ├── comparison_report.md ├── readiness_decision.json ├── readiness_report.html ├── improvement_roadmap.md └── proposal_scope.md
Use regular files only. Do not place symlinks, scripts, secret dumps, or unreviewed production exports in the proof-lab folders. The approved design assumes evidence hashing and snapshot tracking, so clean file hygiene matters.

SECTION 4Exact evidence and case preparation

Evidence preparation

Before you run add-evidence, sort your source files by job outside the managed workspace. The command copies and hashes each selected file.

Suggested source folderPut these files hereWhy they matter
source-evidence/current-process/Current SOP notes, operator checklists, intake screenshots, CSV exports with sensitive data removedShows how the work really starts and finishes today
source-evidence/approved-examples/Completed human-approved outputsDefines what “correct” looks like
source-evidence/policies/Policy notes, contract rules, amount limits, red-line conditionsPrevents invented rules
source-evidence/exceptions/Known exception examples and escalation notesProves exceptions can be detected and owned
source-evidence/recovery/Undo steps, hold procedures, emergency stop notes, recovery drillsRequired for any level above assisted drafting

Case preparation

Each historical case should contain three bounded views of the same work:

  • Input evidence: the files the workflow saw at the time.
  • Expected human outcome: what the approved human result actually was.
  • Proposed system outcome: what the candidate rule or draft system would have done, without external writes.

Cover at least these seven case classes for shadow mode:

Required case classes

  • normal
  • near_limit
  • missing_data
  • duplicate

Also required

  • conflicting
  • expired
  • external_failure

The 11 hard gates

Every gate uses exactly one of pass, fail, or unknown. A score never overrides a failed or unknown gate.

Gate IDPlain meaning
input_identifiableThe system can tell which inputs belong to one job.
completion_testableThere is a repeatable check for “finished correctly.”
standard_policy_confirmedAn accountable human confirmed the normal rules.
exceptions_detectableKnown abnormal cases can be detected before action.
exception_owner_assignedA named person owns cases the system cannot decide.
least_privilege_availableThe future runtime can use only the minimum access needed.
idempotency_definedRetries and duplicates cannot repeat the business action.
recovery_definedHold, undo, and restart steps are written and testable.
kill_switch_ownedA named owner can stop the workflow quickly.
data_use_permittedThe organization permits this data to be used for the test.
shadow_test_passedHistorical comparison met the agreed quality threshold.
Beginner rule. If you only have ten cases, that is enough to learn. It is not enough to claim L3 or L4. In that situation, use the proof lab to deliver a careful assist_only, not_ready, or “keep shadow mode running” result.

SECTION 5The command loop

1

Add evidence. Register each file with a role, provider, and classification so the report can trace every conclusion back to a real source.

ai-automation-kit autopilot-proof-lab add-evidence \
  --workspace .tmp/proof-lab-invoice \
  --source ./source-files/approved-example-001.md \
  --role approved_example \
  --provided-by "Operations owner" \
  --classification confidential
evidence_id=ev-0001
status=present
external_actions_enabled=false
2

Add a historical case. Keep one case ID per real situation. Include input, expected human result, proposed system result, risk tier, and case class.

ai-automation-kit autopilot-proof-lab add-case \
  --workspace .tmp/proof-lab-invoice \
  --case-id case-001 \
  --input ./cases-source/case-001/input.json \
  --expected ./cases-source/case-001/expected.json \
  --proposed ./cases-source/case-001/proposed.json \
  --case-class normal \
  --risk-tier low \
  --expected-route standard \
  --proposed-route standard
case_id=case-001
external_actions_enabled=false
3

Set a hard gate. Record whether a gate passes, fails, or is still unknown. Give the accountable owner and supporting evidence.

ai-automation-kit autopilot-proof-lab set-gate \
  --workspace .tmp/proof-lab-invoice \
  --gate recovery_defined \
  --status fail \
  --owner "Procurement systems owner" \
  --evidence-id ev-0001 \
  --note "No tested vendor deactivation path yet"
gate=recovery_defined
status=fail
external_actions_enabled=false
4

Evaluate. This compares shadow cases, checks hard gates, writes the readiness decision, and generates the human-readable report.

ai-automation-kit autopilot-proof-lab evaluate \
  --workspace .tmp/proof-lab-invoice
status=not_ready
stage=not_ready
decision=not_ready
maximum_level=L0
next_action=Resolve the blocking input or completion definition before automation.
external_actions_enabled=false
5

Check status. Use this after every change. It tells you the current state, next action, and whether any decision has become stale.

ai-automation-kit autopilot-proof-lab status \
  --workspace .tmp/proof-lab-invoice
status=stale_decision
stage=stale_decision
decision=not_ready
maximum_level=L0
next_action=Restore changed evidence or reassess the changed configuration.
external_actions_enabled=false
6

Serve the local browser UI. Open the proof lab on the same computer. Keep it on 127.0.0.1; this is not a public web app.

ai-automation-kit autopilot-proof-lab serve \
  --workspace .tmp/proof-lab-invoice \
  --language en \
  --no-open
http://127.0.0.1:8080/

SECTION 7L0-L4, shadow mode, stale evidence, and honest refusal

L0Not ready. Required evidence, rules, ownership, or recovery is missing.
L1Assisted drafting. AI helps organize evidence and drafts, but human decisions stay.
L2Shadow mode. The proposed workflow runs locally without external writes and is compared against real human outcomes.
L3Conditional autopilot readiness for a future separate runtime. Standard cases only; exceptions stop.
L4Unattended standard-path readiness for a future separate runtime, with quarantine, stop, recovery, and audit controls.

Shadow mode in plain language

Shadow mode means “act as if the workflow were live, but do not change any outside system.” The proof lab compares what the proposed system would have done against what humans actually approved in the past.

Shadow mode must showWhy it matters
Exact or material matchesRoutine cases should line up with reality
False positives and false negativesA high-risk miss cannot be hidden inside a nice average
Duplicate-action handlingReplay safety is mandatory for stronger automation
Exception detectionUnusual cases must be caught and routed to humans
Recovery simulationThere must be a tested way back when something goes wrong

Stale evidence

A decision becomes stale when the evidence snapshot changes. Typical triggers:

  • a source file was edited, replaced, or removed;
  • a policy version changed;
  • a responsible owner changed;
  • a new exception route appeared;
  • a case or proposed policy file changed after the last evaluation.
Stale means stop and reassess. A stale decision cannot justify a pilot, a deployment step, or any external action plan. Re-run status, review the changed evidence, then run evaluate again.

Honest refusal

An honest refusal is a valuable result. The proof lab should clearly distinguish:

  • temporary gap: more evidence or one owned fix can reopen the path;
  • poor economics: automation costs more than the real business value;
  • responsibility boundary: the decision should remain human by policy or law;
  • technical impossibility: there is no safe trigger, no safe input quality, or no safe reversal path.

SECTION 8Side-business delivery, privacy, and safety

How to deliver this as a side-business service

You can sell the assessment work without claiming full automation. A clean beginner delivery usually includes:

  • readiness_report.html for non-technical stakeholders;
  • readiness_decision.json for the exact machine-readable result;
  • comparison_report.md with shadow findings and failure classes;
  • improvement_roadmap.md with owner, action, and acceptance check per gap;
  • proposal_scope.md explaining what stays human and why.
Do not oversell. If the proof lab says not_ready or assist_only, present that as the service value. It prevented a risky promise. It did not “fail.”

Privacy and safety rules

  • Prefer masked or reduced sample data whenever possible.
  • Do not paste secrets, passwords, or raw production tokens into the workspace.
  • Do not place live payment files, live HR exports, or access-grant payloads in public demo folders.
  • Keep the local browser UI on 127.0.0.1 only.
  • Remember that AI suggestions never become production policy until an authorized human confirms them.

SECTION 9Troubleshooting

The command is not found

Activate the project environment again with source .venv/bin/activate, then reinstall the package with python3 -m pip install -e ..

The proof lab says a decision is stale

Run ai-automation-kit autopilot-proof-lab status --workspace .... Review which evidence changed. If the change is intentional, re-run evaluate. If the change is suspicious, stop and inspect the workspace before continuing.

The browser page does not open

Make sure the terminal running ai-automation-kit autopilot-proof-lab serve is still active. Open the exact 127.0.0.1 address printed by the command. If the port is busy, restart the server and use the new printed address.

The result is stuck at L1 or L2

Read the blocking hard gates first. Common blockers are missing recovery, missing duplicate control, missing exception ownership, or too little shadow evidence. The remedy belongs in improvement_roadmap.md.

A client wants “full automation now”

Do not promise it. Show the readiness report, point to the current human-approval boundary, and explain that external actions would require a future separate runtime with new business, legal, and security approval.

Good stopping point. When you feel pressure to promise more than the evidence supports, stop at assist_only, not_ready, or continued shadow mode. That is exactly what this proof lab is for.
