# Autopilot Readiness Builder 人間中心設計

更新日: 2026-07-13
状態: 設計案。実装開始前の確認対象

## 1. この仕組みが解決すること

企業が「この仕事を完全自動化したい」と考えても、必要な資料、判断ルール、例外、権限、失敗時の戻し方が決まっていないことがあります。その状態でAIへ実行権限を渡すと、正常時には動いても、例外時に誤った処理を続ける危険があります。

Autopilot Readiness Builderは、自動化を無理に成立させる道具ではありません。人が現在行っている仕事をAIと一緒に整理し、次の四つを判定する道具です。

1. 今すぐ自動化できる
2. 不足事項を解決すれば自動化できる
3. 人の判断を残せば自動化できる
4. 現在の条件では自動化すべきではない

「できない」という判定は失敗ではありません。事故や過大な導入費用を防いだ、価値のある調査結果として保存します。

## 2. 利用する人

### 自社の業務改善担当者

- 自動化したい仕事はあるが、何を決めればよいか分からない
- 現場担当者の経験を業務ルールとして整理したい
- 本番導入前に、過去データで安全性を確かめたい

### 副業・受託で自動化を提案する人

- 顧客へ聞くべき内容を漏らしたくない
- 実現できない完全自動化を約束したくない
- 調査、試運転、導入判定を成果物として納品したい

### 現場担当者

- 自分の仕事をプログラム用語で説明できない
- AIから一度に大量の質問をされたくない
- なぜ質問されているか理解してから回答したい

## 3. 人間中心の原則

### 一問ずつ聞く

AIは一度に質問票を表示しません。回答によって次の質問を変えます。各質問には、質問理由、回答例、「分からない」の選択肢を付けます。

### 現場の言葉から始める

「冪等性」「補償トランザクション」などの言葉を最初から使いません。

- 「同じ資料が二回来たらどうしますか」
- 「間違って登録した場合、元へ戻せますか」
- 「この判断を普段は誰が行っていますか」

と質問し、内部で技術要件へ変換します。

### AIは決定ルールの候補を作る

過去資料からルール候補を抽出しても、AIだけで正式ルールにしません。現場担当者と責任者が確認したものだけを有効にします。

### 不明を不明のまま保存する

推測で質問を埋めません。不明事項には担当者、確認期限、次の確認方法を付けます。

### 中止理由を具体的に伝える

「安全性が不足しています」だけでは役に立ちません。

> 取引先の銀行口座を誤登録した場合の取消手順がありません。経理責任者と、登録取消および支払停止の手順を決めるまで、取引先登録の完全自動化は開始できません。

のように、止めた理由と解決方法を示します。

## 4. 利用者が進む道筋

```text
目的を選ぶ
  ↓
現在の作業を説明する
  ↓
資料を指定フォルダーへ入れる
  ↓
AIが資料と過去の完成物を確認する
  ↓
AIが一問ずつ不足事項を聞く
  ↓
判断ルールと例外を人が確認する
  ↓
過去データで監視運転する
  ↓
人の結果とAIの結果を比較する
  ↓
完全自動・条件付き・半自動・中止を判定する
  ↓
判定理由と解決ロードマップを保存する
```

## 5. 最初に選ぶ目的

入口では、利用者へシステム構成を選ばせません。

- 自社の仕事を楽にしたい
- 顧客へ自動化を提案したい
- すでにある半自動処理を無人化したい
- 自動化できるかだけ調べたい

目的によって、説明、質問、生成する資料を変えます。

## 6. AIが確認する九つの領域

### 6.1 仕事の開始

- 何が起きたら仕事を始めるか
- 誰が開始を判断しているか
- 日時、ファイル到着、申請登録など、機械で検出できるか
- 同じ依頼が二回来る可能性があるか

### 6.2 入力資料

- 必須資料と任意資料
- 保存場所とファイル形式
- 必須項目
- 正しい資料と古い資料の見分け方
- 個人情報、機密情報、法定保存資料の有無

### 6.3 完成物

- 正しい完成例
- 完成物の保存場所
- 必須項目とファイル名
- 完了と判断する条件
- 誰が完成物を利用するか

### 6.4 判断ルール

- 数値、期限、状態で決められる条件
- 優先順位
- 許可・拒否条件
- 人の経験に依存している判断
- 法律、契約、社内規程に基づく判断

### 6.5 例外

- 普段と違う案件
- 情報不足や矛盾
- 金額や件数の上限超過
- 顧客苦情、安全、人事、法務、情報セキュリティに関する案件
- 例外を引き受ける人と期限

### 6.6 外部操作と権限

- 読み取りと書き込みの区別
- 操作するシステム
- 必要最小限の権限
- 自動化専用アカウント
- 実行回数、金額、件数の上限

### 6.7 失敗と復旧

- 一時的な失敗と恒久的な失敗
- 再試行回数
- 二重実行の防止
- 失敗案件の隔離場所
- 取消方法
- 取消できない操作
- 緊急停止責任者

### 6.8 責任と規則

- 業務責任者
- システム管理者
- データ管理責任者
- 法務、経理、人事、セキュリティ等の確認要否
- 保存期間
- 説明・異議申立て・再確認の方法

### 6.9 効果

- 現在の処理時間
- 月間件数
- 間違いと手戻り
- 自動化後の目標
- 導入費用と維持費用
- 自動化しない場合との比較

## 7. 必須ゲート

点数が高くても、次の必須条件を満たさなければ完全自動化を許可しません。

| ゲート | 合格条件 | 不合格時 |
|---|---|---|
| 入力 | 必須資料と正しい形式を判定できる | 資料整備へ戻す |
| 完成 | 正しい結果と完成条件が明確 | 完成例を追加する |
| 判断 | 標準案件の決定ルールが明確 | 半自動へ下げる |
| 例外 | 例外の検出条件と担当者が明確 | 本番化しない |
| 権限 | 必要最小限の専用権限を用意 | 接続しない |
| 二重防止 | 同一案件を識別できる | 自動書込み禁止 |
| 復旧 | 取消または補償方法がある | 無人実行禁止 |
| 停止 | 緊急停止方法と責任者が明確 | 本番化しない |
| 規則 | 法令・契約・社内規程上の許可 | 中止または専門確認 |
| 試験 | 過去データと監視運転で基準達成 | 改善または中止 |

## 8. 判定レベル

### L0: 自動化不可

必要資料、責任者、判断ルール、復旧方法などが不足しています。中止理由と、再検討に必要な行動を出力します。

### L1: AI補助

AIが資料整理、不足確認、下書き作成を行います。現在のHuman Approval Editionに相当します。

### L2: 監視運転

AIは本番と同じ判断を行いますが、外部システムを変更しません。人の実際の結果と比較します。

### L3: 条件付き自動化

標準案件だけを自動処理します。上限超過、情報不足、矛盾、高リスク案件は人へ戻します。

### L4: 無人運転

決定ルール内の標準案件を無人処理します。停止、隔離、復旧、監査が常に有効です。「例外が一切ない」という意味ではありません。

L3とL4は、将来の別Autopilot実行サービスへ進める準備が整ったという診断結果です。現在の22パックをそのまま無人実行する設定ではありません。現行プロジェクトでは全パックの毎回承認を維持し、L3/L4判定が出ても外部操作は実行しません。

将来、毎回承認を不要にする場合は、企業責任者が事前承認した決定ルール、操作上限、例外、停止、復旧を含む別の実行契約を作り、法務・セキュリティ・業務責任者による個別確認を行います。診断結果だけで現行パックの承認要件を解除することは禁止します。

## 9. 判定の考え方

AIが最終判定を一つの自信度だけで決めることは禁止します。

1. 必須ゲートを機械的に確認する
2. 根拠資料を各判定へ紐付ける
3. 不足・矛盾・推測を分離する
4. 高リスク操作を別判定する
5. 過去データの比較結果を確認する
6. 企業の責任者が自動化範囲を確認する

AIは推奨レベルと理由を提示します。規則や責任の確定は企業側が行います。

### 既存パックの禁止操作は解除できない

診断結果や企業独自設定は、22パックに登録されている禁止操作と承認要件を狭めることはできても、解除・上書きできません。例えば取引先登録パックの銀行情報更新、アクセス確認パックの権限付与などは、診断点数が高くても現行パック内では自動許可しません。将来それらを扱う場合は、別の実行サービス、個別の法務・セキュリティ確認、専用の操作契約が必要です。

## 10. 「無理」の種類

### 現時点では無理

資料やルールを整えれば再検討できます。

### 費用対効果が合わない

技術的には可能でも、件数が少ない、例外が多い、保守費用が大きい場合です。

### 責任上無理

人事、契約、金銭、アクセス権、法令等で人の判断を残す必要があります。下準備だけを自動化します。

### 技術的に無理

接続手段がない、入力品質を判定できない、取消不能な操作しかない場合です。

### 企業として許可できない

社内規程、顧客契約、データ保管条件等に反する場合です。

## 11. 解決ロードマップ

不合格項目ごとに、AIは次を生成します。

- 課題
- なぜ必要か
- 解決担当者
- 必要資料
- 解決手順
- 完了条件
- 確認期限
- 解決後に到達できる自動化レベル

例:

```text
課題: 同じ申請を識別する番号がない
危険: 再試行時に二重登録される
担当候補: 業務責任者、システム管理者
解決案: 申請番号と対象月を組み合わせた一意キーを定義する
完了条件: 過去100件で重複がなく、再実行試験で二重登録されない
到達可能レベル: L3 条件付き自動化
```

## 11.1 監視運転の最低基準

企業ごとに合格基準を追加できますが、L3またはL4を提案するには最低でも次を満たします。

- 通常、上限付近、情報不足、重複、矛盾、期限超過、外部障害の各ケースを含む
- 対象期間の業務量が少ない場合を除き、最低100件または直近3か月分の多い方を試す
- 高リスク業務は、重大な見逃しを0件とする
- 例外条件ごとに少なくとも1件のテストを用意する
- 古い資料、変更された資料、欠損資料で停止できることを確かめる
- 同じ案件を再実行しても二重処理されないことを確かめる
- 取消または補償処理を少なくとも1回実演する
- 低・中・高リスクの結果を分けて記録する

100件を用意できない場合は、自動的に合格にせず、L2監視運転を延長するか、責任者が少数データの理由と追加監視期間を記録します。

## 11.2 判定の有効期限と再診断

次の変更が検出された場合、以前の合格判定を`判定失効`に変更し、再診断へ戻します。

- 入力資料または必須項目
- 判断ルール、金額上限、例外条件
- パック定義または禁止操作
- 接続先、権限、認証方法
- 法令、契約、社内規程
- AIモデルまたは主要な抽出処理
- 責任者、停止担当者、復旧方法

資料ハッシュとルール版が一致しない判定は、本番許可に再利用できません。

## 11.3 人が確認した記録

ルール、ゲート、最終判定の確認記録には、確認者名だけでなく次を保存します。

- 確認者の役割
- 確認日時
- その人が確認権限を持つ根拠
- 使用したパックmanifestのハッシュ
- 使用した資料スナップショットのハッシュ
- 直前の判定ハッシュ
- 確認理由と制限事項

## 12. 画面設計

### ホーム

「自動化できるか調べる」「途中から再開」「判定結果を見る」を表示します。

### 一問一答

質問、理由、回答例、資料を開くボタン、「分からない」を表示します。

### 資料準備

必要資料、用意済み、不足、利用不可、担当者確認中を色とアイコンだけに頼らず文字で表示します。

### 現在の作業フロー

人が行っている順番、利用資料、判断、例外、完成物をフロー図で表示します。

### 判断ルール

条件、結果、例外、根拠、確認者を表で表示します。AIが抽出した未確認ルールには「候補」と表示します。

### 試運転比較

人の結果、AIの結果、差異、原因、修正内容を並べます。

### 判定

自動化レベル、合格ゲート、不合格ゲート、リスク、推奨、次の行動を表示します。

### 運転準備

本番化する場合だけ、権限、上限、停止、復旧、監視担当者を最終確認します。

## 13. 生成する成果物

- `current_workflow.md`: 現在の人の作業
- `automation_scope.md`: 自動化する範囲と人に残す範囲
- `source_inventory.csv`: 入力資料一覧
- `decision_table.csv`: 判断ルール
- `exception_register.csv`: 例外条件と担当者
- `connector_permissions.md`: 接続先と必要権限
- `recovery_plan.md`: 再試行、取消、補償
- `kill_switch_plan.md`: 緊急停止
- `shadow_test_cases.csv`: 監視運転テスト
- `comparison_report.md`: 人とAIの比較
- `readiness_decision.json`: 機械可読の判定
- `readiness_report.html`: 人向け判定結果
- `improvement_roadmap.md`: 不足事項の解決手順
- `proposal_scope.md`: 顧客へ提示する導入範囲

## 14. 22パックへの適用

22パックすべてを診断対象にします。ただし、パック名だけで自動化レベルを決めません。企業ごとの資料、判断ルール、操作範囲、法令、復旧方法で判定します。

- 月報や経営要約は、保存までの無人化候補
- 問い合わせ、営業、プロジェクト、会議、社内依頼、引き継ぎは、低リスクな登録や振り分けの条件付き自動化候補
- 財務、勤怠、経費、請求、契約、発注、人事、法令、アクセス、助成金等は、準備を無人化し、重要判断を別ゲートにする候補

## 15. 本番化の順序

1. 匿名化・サンプル資料で診断
2. 過去資料で再現
3. 人の結果と比較
4. 読み取り専用で監視運転
5. 低リスク操作だけを少量実行
6. 件数・範囲を段階的に拡大
7. 定期的にルールと権限を再確認

一度合格したら永久に合格ではありません。資料形式、規則、担当者、接続先、モデルが変わった場合は再診断します。

## 16. 販売時の約束

販売文では「何でも完全自動化」と表現しません。

> 標準案件を自動化し、判断できない案件を止めて人へ戻す仕組みを、企業の資料とルールに合わせて構築します。

と説明します。

調査の結果がL0でも、現在のフロー、不足事項、改善ロードマップ、費用対効果を納品します。これにより、実装しない判断にも価値を持たせます。

## 17. 実装前の確認事項

- 現行のHuman Approval Editionは安全な別製品として維持する
- 外部操作を行うAutopilotは別リポジトリまたは明確に分離した実行サービスとする
- 最初の実装範囲は診断、成果物生成、監視運転までとする
- 外部書込みは診断基盤と監視運転が検証された後に設計する
- 高リスク業務の判断をAIの自由文だけで許可しない

## 18. 成功条件

この設計が成功したと言えるのは、完全自動化の件数が多いときではありません。

- 利用者が次に用意すべきものを理解できる
- 不足資料や曖昧な判断を見逃さない
- 無理な案件を理由付きで止められる
- 解決可能な課題には具体的な道筋を出せる
- 人の結果とAIの結果を比較できる
- 自動化範囲と人の責任が明確になる
- 導入しない判断も成果物として説明できる

この基準を、今後のAutopilot設計と実装の中心思想とします。
