Phase 1A local workflow

Monthly operator workspace manual

This manual shows the full monthly office workspace flow for a beginner: install, confirm Codex login, ask Codex to create the workspace, place three file types, inspect, answer, generate, cancel or retry, approve, and roll forward to the next month.

Honest release scope

This release is one monthly office workspace for a monthly report pack. It is local-only and does not replace the older flow-project path.

Platform truth

macOS or Linux only for workspace mutation. Windows is fine for reading docs, but not for the safe create and approval-mutation path in this release.

Safety truth

No API key is required. No external sending occurs. Approval is a local record on this device.

The manual below is written for the operator who is about to run the first real local month. Keep it open while you work. If the workspace is already created, jump straight to the folder map and the inspect step.

Install and confirm Codex login

Install the package in a local Python environment first, then confirm that Codex login is already available on this machine. This flow depends on the local Codex login state, not on pasting credentials into chat.

Install the kit in the repository checkout

This keeps the CLI and bundled monthly-report pack together on the same machine as the local files you review.

python3 -m venv .venv
source .venv/bin/activate
python3 -m pip install --upgrade pip setuptools
python3 -m pip install -e .
Confirm Codex login before setup

The login check must succeed before the workspace is created for a guided Codex-led setup session.

codex login status
Stop here if login is not ready

If codex login status fails, let the human finish the login flow first. Do not switch to an API-key explanation. Do not claim that a remote send mode will fix it.

Ask Codex to set up the workspace

Paste the opening message from START_WITH_CODEX.md into Codex. That file tells Codex to ask one question at a time and confirm the save root, workspace name, approver, and first month before running the CLI.

Exact setup command

Codex should eventually run ai-automation-kit office-workspace create with the confirmed values, then validate with ai-automation-kit office-workspace status --workspace.

Do not improvise flags

Do not add custom executable paths, shell bypasses, approval bypasses, remote-send options, or unsafe shortcut flags. The setup path is intentionally narrow.

ai-automation-kit office-workspace create \
  --root "./workspaces" \
  --name "Construction Monthly" \
  --approver "Operations Manager" \
  --pin "482913" \
  --period "2026-07" \
  --language en

ai-automation-kit office-workspace status \
  --workspace "./workspaces/Construction_Monthly" \
  --json

Single-line references for copy and search: ai-automation-kit office-workspace create --root ./workspaces --name "Construction Monthly" --approver "Operations Manager" --pin 482913 --period 2026-07 --language en and ai-automation-kit office-workspace status --workspace ./workspaces/Construction_Monthly --json.

The three file types

Every beginner should understand the three file types before the first inspect step. The AI works better when the folders are boring and explicit.

Folder What goes there Concrete example
01_APPROVED_PAST_OUTPUTS Older approved monthly reports that show tone, structure, and naming. 2026-05-monthly-report.md, 2026-06-monthly-report.docx
02_PAST_SUPPORTING_FILES Older supporting references that help explain recurring metrics, field names, or background notes. vendor-summary.xlsx, finance-notes.md, policy-export.pdf
03_CURRENT_INPUTS/<YYYY-MM> This month's files only. Put the current month in the dated folder and keep unrelated files out. 2026-07/labor_hours.csv, 2026-07/incident-notes.md, 2026-07/cost-tracker.xlsx
Keep the three file types separate

Do not drop this month's raw files into 01_APPROVED_PAST_OUTPUTS. Do not treat an old draft as an approved style example. The separation is part of the safety model.

Inspect the month

Once the files are in place, run the local inspect step. This confirms which files were accepted and whether the next action is a question or a ready-to-run draft request.

ai-automation-kit office-workspace inspect \
  --workspace "./workspaces/Construction_Monthly" \
  --period "2026-07"

Single-line inspect reference: ai-automation-kit office-workspace inspect --workspace ./workspaces/Construction_Monthly --period 2026-07.

  • Use inspect after every material change that matters.
  • Inspect does not send anything. It writes local state and a source manifest.
  • If the stage becomes questioning, the next step is to answer, not to force generation.

Answer one question at a time

The workspace asks only the missing required question. Keep the answer short, factual, and local to the month you are drafting. The question can be answered from the localhost UI after the inspect step.

Good answer

Audience: Operations board and finance lead. This tells the draft how direct or formal it should be.

Bad answer

A vague answer such as make it good forces the draft toward guesswork and usually causes revision churn.

Once the answer is saved, the current period moves to ready_for_run. At that point the draft can be generated from the UI.

Generate the draft

Generation is still local. Codex reads a safe snapshot inside a disposable run directory and writes only a staged draft that the host validates before promoting it into 05_DRAFTS.

What appears after generation

The period moves to review, a draft file appears in 05_DRAFTS/<YYYY-MM>, and the UI shows provenance and unresolved items for the human reviewer.

Use the folder icon or local finder shortcut to open the draft. Read it as a draft, not as an approved final statement.

Single-line server reference: ai-automation-kit office-workspace serve --root ./workspaces --language en --no-open.

Cancel and retry safely

If the wrong files were used, if a question was answered badly, or if the run should stop, cancel the current run and correct the local state before retrying. The workspace keeps prior approved outputs and accepted inputs safe.

When to cancel

  • The current month folder contains the wrong month.
  • The answer changed after a stakeholder review.
  • The run stalled or produced a missing-information question that needs a file fix.

What to do next

  • Fix the folder contents or answer.
  • Inspect again if the materials changed.
  • Retry the draft generation for the same month.

Approve the local copy

Approval moves a reviewed local draft into 06_APPROVED_OUTPUTS and writes the approval trail into 07_AUDIT. The human approver name and the configured PIN are part of the approval step.

05_DRAFTS

Read the draft here, compare it with the accepted inputs, and confirm wording before approval.

06_APPROVED_OUTPUTS

The approved local copy is saved here and gets its own hash.

07_AUDIT

The audit trail stores the local approval chain so the next month can cite the approved reference honestly.

Exact approved hash matters

The approved hash lets the next period cite a style reference without pretending that a later edit was already reviewed. Keep the hash and the audit record together.

Prepare the next month

After approval, create the next month and choose whether the approved prior output should become a style reference. Rollover is append-only. It does not delete the prior period.

  • Rollover keeps the old month intact.
  • The approved prior month can be referenced only after approval exists.
  • The next month starts empty in the current-input folder so the new files stay separate.

What AI can do and what a human must decide

What AI can do
  • Read the accepted local snapshot for the current month.
  • Draft wording, outline sections, and point at missing facts.
  • Reuse the approved style reference only when the host confirms it.
  • Prepare a local draft for review.
What a human must decide
  • Which files are approved to use.
  • Whether the audience, numbers, and wording are correct.
  • Whether to cancel, retry, approve, or roll forward.
  • Any external submission after the local review is complete.
What the local system guarantees
  • No API key is required.
  • No external sending occurs.
  • Original input files remain untouched.
  • Approved copies and audit records stay visible on disk.
Local-only workflow

This manual is intentionally narrow. It does not promise email sending, Slack posting, publication, payment, or production database updates. Those are outside the scope of this office workspace.

Troubleshooting

What happened What it usually means What to do next
Codex login check failed The local Codex session is not ready on this machine. Finish the local login flow, rerun codex login status, then resume setup.
Workspace creation stopped on Windows The safe create and approval-mutation path is not supported there in Phase 1A. Use a macOS or Linux machine for creation and approval mutation.
Inspect found no useful files The current month folder is empty, or the files were placed in the wrong directory. Recheck 01_APPROVED_PAST_OUTPUTS, 02_PAST_SUPPORTING_FILES, and 03_CURRENT_INPUTS/<YYYY-MM>.
The period stays in questioning A required answer is still missing. Answer the question from the localhost UI, then try generation again.
The draft should not be trusted yet The period is ready for review, not approved. Read the draft in 05_DRAFTS, compare it with the local source files, then approve the local copy.
The UI says manual recovery is required The workspace state, run directory, or lock file needs a human check. Inspect 00_START_HERE/workspace_status.json, .system/workspace.json, .system/periods/<YYYY-MM>/state.json, .system/runs/, and 07_AUDIT/audit.jsonl before deciding the next action.
What to do when you must pause the month

Leave the accepted files where they are, write down which answer still needs confirmation, and stop before approval. A pause is safer than approving uncertain wording.

What to do when the next month should reuse style carefully

Approve the prior month first, then confirm the style reference during rollover. Do not treat an unapproved draft as a style baseline.
